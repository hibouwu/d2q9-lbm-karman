### 超订阅问题说明

`np=2 OMP=16` 和 `np=4 OMP=8` 的总线程数 np×OMP=32，恰好等于逻辑核数。由于 SMT 开启，16 个物理核各对应 2 个逻辑核，填满逻辑核会导致不同 rank 的线程争抢同一物理核的执行资源，性能断崖（36–68 MLUPS vs 正常 ~780 MLUPS）。

这两个配置的数字**不反映算法性能**，不得写入报告正式表格。

### 后续计划

未来 benchmark 应在**禁用 SMT**（关闭超线程）的条件下运行，使逻辑核数等于物理核数（16），避免超订阅风险：

```bash
# 临时禁用 SMT（需 root，重启后恢复）
echo off | sudo tee /sys/devices/system/cpu/smt/control
# 验证：nproc 应输出 16
nproc
# 恢复
echo on | sudo tee /sys/devices/system/cpu/smt/control
```

禁用 SMT 后，bench.sh 的配置矩阵中 np×OMP ≤ 16 的配置均有效，可以增加 np=4 OMP=4（=16）作为扩展性参考。
