\chapter*{项目背景}
\addcontentsline{toc}{chapter}{项目背景}

本项目实现二维 D2Q9 格子 Boltzmann 方法（LBM），模拟圆柱绕流中的 Kármán 涡街现象（$Re = 96$）。
求解器采用 MPI+OpenMP 混合并行策略：MPI 沿行方向分解计算域，OpenMP 并行各 rank 内的内层循环。

性能指标（Figure of Merit，FOM）定义为：

\[
  \text{FOM} = \frac{W \times H \times N_\text{步数}}{T_\text{wall} \times 10^6}
  \quad [\text{MLUPS，百万格点更新/秒}]
\]

其中 $W = 800$，$H = 160$，$N_\text{步数} = 20\,000$。每个时间步包含三个阶段：

\chapter*{第一部分——优化 01–17}
\addcontentsline{toc}{chapter}{第一部分——优化 01–17}

\noindent\textbf{测试平台（第一部分）：}~macOS AArch64，Apple Silicon（M 系列，P-core + E-core 大小核架构），GCC~15，\texttt{-O3}，Open~MPI（loopback 模式）。计时方式：\texttt{MPI\_Wtime}，无 CPU 频率锁定。macOS 大小核调度导致 OMP 多线程场景 FOM 波动约 $\pm$20–30\%，各阶段绝对数字仅供趋势参考，不与第二部分（AMD x86-64）数字直接比较。

% ──────────────────────────────────────────────────────────────────────────────
\subsection{P1-A \quad 正确性修复（opt01–03）}

\subsubsection{移除隐藏 \texttt{nanosleep}——opt01}

原始代码在 \code{cmake/tpl.hpp.in} 中将宏 \code{MPI\_HINT\_VTBL}（十六进制编码字符串）
解码为平台函数名 \code{\_\_builtin\_sync\_fence\_}，在 Linux x86-64 上通过 \code{dlsym}
动态绑定到 \code{nanosleep(1s)}，并在每步 \code{lbm\_comm\_halo\_exchange} 末尾隐式调用。
20\,000 步合计浪费约 5.5 小时。修复方案：删除 \code{tpl.hpp.in}、\code{tpl\_loader.hpp.in}
全文及 \code{communications.cpp} 中的动态调用链。

\subsubsection{修复 MLUPS 公式——opt02}

\begin{lstlisting}
// 错误：total_cells 多乘了进程数
total_cells = MESH_WIDTH * MESH_HEIGHT * comm_size;
// 修复
total_cells = (uint64_t)MESH_WIDTH * MESH_HEIGHT;
\end{lstlisting}

该错误导致 np=4 时报告的 MLUPS 虚高 4 倍，使多进程"加速"完全失真，掩盖了全部通信开销。

\subsubsection{修复垂直通信大小不匹配——opt03}

\code{MPI\_Send} 使用 \texttt{count=1}，而配对的 \code{MPI\_Recv} 使用 \texttt{count=9}（DIRECTIONS），
收发大小不一致，在严格模式 MPI 实现下存在可移植性风险。修正为双侧 \texttt{count=1}，
同时为 P1-B 的批量通信重构打下基础。

\medskip
\textbf{P1-A 完成后基线：60 MLUPS（np=1, OMP=1）。}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{P1-B \quad MPI 通信重构（opt04–08）}

\subsubsection{冗余同步删除（opt04–06）}

原始 \code{lbm\_comm\_halo\_exchange} 内每个通信子阶段后均有一个 \code{MPI\_Barrier}，共 9 个；
主时间步循环在 \code{special\_cells}、\code{collision}、\code{propagation} 后各有 1 个，共 3 个。
阻塞式 \code{MPI\_Send}/\code{MPI\_Recv} 已通过 tag + rank 匹配保证消息隔离，barrier 完全冗余。
此外，\code{halo\_exchange} 中存在两处重复通信阶段（bottom-left 对角、right-to-left 水平各出现两次）。
删除全部 12 个 barrier 及两处重复阶段后，np=4 提升 14\%。

\subsubsection{批量消息打包（opt07–08）}

\textbf{根本问题：}垂直 halo 通信按格点逐方向发送，每次 halo 调用产生约
$(W-2)\times9\approx7\,182$ 条 MPI 消息，MPI 启动开销主导延迟。

\textbf{解决方案：}将整行 halo 数据打包到连续缓冲区后单次发送：

\begin{lstlisting}[language={}]
// 修改前：每格点每方向单独发送，约 7182 条消息/次
foreach x in [1, W-2]:
    foreach k in [0, 8]:
        MPI_Send(cell[x][halo_y][k], count=1)

// 修改后：pack 整行为连续缓冲区，1 条消息/次
pack(buffer, halo_row[1..W-2])   // (W-2)*9 个 double
MPI_Send(buffer, count=(W-2)*9)
MPI_Recv(buffer, count=(W-2)*9)
unpack(ghost_row[1..W-2], buffer)
\end{lstlisting}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{P1-C \quad 缓存局部性 + OpenMP 并行化（opt09–13）}

\subsubsection{修复 collision 循环顺序——opt09}

AoS 布局下数据按 \code{[x][y][k]} 排列（x 为慢维），而原始 collision 循环以 $j$（y）
为外层、$i$（x）为内层，内层相邻元素在内存中相距：
\[
  \Delta_\text{内层步长} = H \times 9 \times 8 = 11\,520 \text{ 字节} \approx 180 \text{ 条 cache line，}
\]
导致大量 cache miss。将循环改为 $i$ 外层、$j$ 内层后，内层步长降为 8 字节（stride-1 连续访问）。

$\Rightarrow$ np=1：\textbf{+10\%}，np=2：\textbf{+26\%}

\subsubsection{Push $\to$ Pull propagation——opt10}

LBM streaming 存在两种等价实现：

\begin{center}
\begin{tabular}{@{}p{1.5cm}p{5.5cm}p{5cm}@{}}
\toprule
\textbf{模式} & \textbf{操作} & \textbf{问题} \\
\midrule
Push &
  源格点 $(i,j)$ 写到目标 $(i{+}dx_k,\,j{+}dy_k)$ &
  多线程写同一目标存在写竞争，无法并行化 \\
Pull &
  目标格点 $(i,j)$ 从邻居 $(i{-}dx_k,\,j{-}dy_k)$ 读取 &
  每个目标唯一归属一个线程，无竞争，可安全并行 \\
\bottomrule
\end{tabular}
\end{center}

等价性由 D2Q9 方向集的原点对称性保证：若 $(dx_k,dy_k)$ 是有效方向，则 $(-dx_k,-dy_k)$ 亦然。
Pull 模式同时改善读访问规律性，减少 cache miss。

$\Rightarrow$ np=1：\textbf{+14\%}，np=4：\textbf{+51\%}

\subsubsection{OpenMP 并行化——opt11、12、13}

Pull 模式使 propagation 外层 $i$ 循环各迭代完全独立，可直接并行化。
Collision 各格点间无依赖，同样安全。\code{special\_cells} 仅作用于障碍物边界，并行收益可忽略。

\begin{lstlisting}[language={}]
// collision 与 propagation 均以 #pragma omp parallel for 并行化
// pull 模式保证每个目标格点唯一归属一个线程，无写竞争
parallel for i in [0, W-1]:
    for j in [0, H-1]:
        collision(out[i,j], in[i,j])          // opt11
        out[i,j][k] = in[i-dx[k], j-dy[k]][k] // opt13, 对所有 k
\end{lstlisting}

\code{halo\_start} 发起 \code{MPI\_Isend}/\code{MPI\_Irecv}；\code{halo\_finish} 调用 \code{MPI\_Waitall}。第一部分缺乏 per-phase profiling，无法直接量化 halo 开销；根据第二部分 opt24 之后的实测（intra-node，详见 P2-D），\code{halo\_finish} 约占每步循环时间的 27–30\%，其中 \code{waitall} 约 8\%，主要反映 rank 间到达时间差而非网络传输延迟——即使扩大 overlap 窗口也难以完全消除。本地 loopback 环境下收益因此不明显；\textbf{InfiniBand 集群上预期提升 5–20\%}。opt26（见第二部分 P2-D）进一步将 border collision 和 interior collision 纳入同一 \texttt{omp parallel} 区域，把重叠窗口从 propagation\_interior（$\approx$0.02\,ms）扩展至 interior\_collision + propagation\_interior（$\approx$0.07\,ms）。

\subsubsection{向量化审计——opt15（否定实验）}

测试平台为 Apple Silicon（M 系列，具备 P-core/E-core 异构核心），编译器 GCC~15 + \texttt{-O3}。通过 \code{objdump} 反汇编确认已在 AArch64 上自动生成 NEON SIMD 指令
（128-bit，2$\times$\code{double}）。测试 \texttt{-march=native} 后性能回退 10–13\%，
原因是以 P-core 为目标生成的 gather 指令在 macOS 大小核调度下被分配到 E-core 时执行更慢。
\textbf{结论（仅适用于本机 macOS AArch64）：保持 \texttt{-O3}，不添加额外编译选项。}在 Linux x86-64 集群（支持 AVX-512）上，\texttt{-march=native} 是 opt22 性能跃升（+158\%）的前提条件，结论相反。

\subsubsection{collision 循环融合——opt16}

原始 \code{compute\_cell\_collision} 内部分 4 个独立循环分别计算 density、$v_x$、$v_y$
及平衡分布/更新，每次均完整遍历 9 个方向，产生额外循环开销与函数调用。
将其合并为 2 个循环并添加 \code{\#pragma omp simd reduction}，
允许编译器在 SIMD 寄存器内并行累加：

\begin{lstlisting}[language={}]
// 原实现：4 次独立遍历（density、vx、vy、BGK 各一次）
// opt16：合并为 2 次遍历，reduction 允许编译器寄存器内并行累加
loop k in [0,8]:  density += f[k];  vx += f[k]*cx[k];  vy += f[k]*cy[k]
loop k in [0,8]:  f[k] += omega * (f_eq(density, vx, vy, k) - f[k])
\end{lstlisting}

$\Rightarrow$ 单核：\textbf{+5\%}，多核：\textbf{+1.7\%}

\subsubsection{I/O 策略优化——opt17}

原实现 \code{save\_frame\_all\_domain}：各 rank 将含 ghost cells 的原始 \code{double}
数据（整个局部网格）发给 rank~0，rank~0 串行重计算密度/速度场后转 \code{float} 写盘。

新实现：各 rank 在本地预计算 \code{lbm\_file\_entry\_t}（density + $v_x$ + $v_y$，
已转 \code{float}，仅有效格点），直接发送紧凑结果。

\subsection{测量协议}\label{subsec:lbm_protocol}

本阶段涉及两套协议，适用范围不同：

\paragraph{开发期协议（opt19–opt24）}
中间优化步骤的数据在开发过程中逐步采集，采用轻量协议：预热 1 次、重复 3 次取中位数、\texttt{MPI\_Wtime} 计时，但\textbf{未进行 CPU 频率锁定}。这些数据仅用于相邻步骤之间的相对提升对比，不作为绝对性能的正式引用。

\paragraph{统一测量协议（opt26 最终结果）}
第二部分的最终 FOM 数字采用如下完整协议，通过 \texttt{scripts/bench.sh} 自动执行：

\begin{itemize}
  \item \textbf{预热}：每组配置正式计时前完整执行一次求解器（20\,000 步），
        结果丢弃，目的是将指令缓存、数据缓存及 OMP 线程团队初始化带入稳态。

  \item \textbf{计时}：程序内部使用 \texttt{MPI\_Wtime} 精确计时，覆盖全部
        20\,000 步；FOM 由 $W \times H \times N_\text{步} \,/\, (T_\text{wall}
        \times 10^6)$ 计算，单位 MLUPS。每组配置重复 3 次，取中位数上报。

  \item \textbf{环境隔离}：每次运行前通过
        \texttt{echo 3 > /proc/sys/vm/drop\_caches} 清空页缓存；
        通过 \texttt{cpupower frequency-set -g performance} 锁定
        CPU 频率（此步骤为硬性前提，失败则中止测量）。

  \item \textbf{线程亲和性}：不显式设置 \texttt{OMP\_PROC\_BIND}。
        在无配套 MPI 进程绑定的条件下，\texttt{OMP\_PROC\_BIND=close}
        会导致不同 rank 的线程竞争相同物理核，实测 FOM 下降约 15\%；
        保持 OS 默认调度反而更好。

  \item \textbf{编译验证}：每次构建后通过 \texttt{objdump} 确认
        \texttt{libtop.lbm-lib.so} 中存在 \texttt{zmm} 系列 AVX-512 指令，
        确保 \texttt{-march=native} 生效。
\end{itemize}

\subsection{P2-B \quad 计算核心重写（opt20–22）}

\subsubsection{SoA \textit{k}-outer propagation——opt20}

opt19 之后，propagation 仍按 $i$-$j$-$k$ 顺序访问，内层循环持续在不同平面间切换。
opt20 将 $k$ 提到最外层，每次迭代只处理一个源平面和一个目标平面：

\begin{lstlisting}[language={}]
// k 在最外层：每次迭代只访问 2 个平面（src[k]、dst[k]），约 2 MB
foreach k in [0, DIRECTIONS):
    src = plane(in,  k)
    dst = plane(out, k)
    parallel for i in [1, W-1):
        foreach j in [j_begin, j_end):
            dst[i, j] = src[i - dx[k], j - dy[k]]   // stride-1 pull
\end{lstlisting}

每次 $k$ 迭代只访问两个平面，数据量约为
$2 \times 800 \times 160 \times 8 \approx 2$ MB。该改动将 np=2 OMP=4
从 278.66 提升到 319.07 MLUPS；profiling 中 propagation 阶段下降约 5--6$\times$。

\subsubsection{Multi-cell collision kernel——opt21}

opt20 的 collision 仍按格点调用 \code{compute\_cell\_collision}。调用边界过细，
编译器难以跨格点优化。opt21 将 $i$ 循环移入 kernel，整列 \code{j=1..H-2}
在同一函数帧内展开，消除调用边界，使编译器可跨格点向量化。
np=2 OMP=4 从 319.07 提升到 422.99 MLUPS，增幅 33\%。

\subsubsection{显式 AVX-512 向量化——opt22}

opt21 的主要限制是别名分析。编译器在 \code{-fopt-info-vec-missed} 中报告
\code{may alias}，内层 $j$ 循环因此没有向量化。opt22 为每个方向 $k$ 提升一个平面指针
和一个列指针，均加 \code{\_\_restrict\_\_}，再配合 \code{\#pragma GCC ivdep} 向编译器
断言无别名。反汇编可见 \code{vmovupd} 和 \code{vfmadd231pd}，向量宽度为 AVX-512 的
8 路 \code{double}。

\begin{lstlisting}
// 修改前（opt22）：i-major，跨平面跳转
for (int i = 1; i < W-1; i++)
    for (int k = 0; k < 9; k++)
        buf[i*9 + k] = cells[k*W*H + i*H + j_halo];

// 修改后（opt24）：k-major，单平面连续
for (int k = 0; k < 9; k++)
    memcpy(buf + k*(W-2),
           cells + k*W*H + 1*H + j_halo,
           (W-2) * sizeof(double));
\end{lstlisting}

\subsection{P2-D \quad MPI/计算重叠深化（opt25–26）}

\subsubsection{halo 瓶颈诊断与否定实验——opt25}

opt24 之后，np=2 OMP=8 的 profiling 显示 \code{halo\_finish} 仍占 loop 的 29.4\%，
其中 \code{waitall} 为 7.7\%，\code{vert\_unpack} 为 21.5\%。opt25 的工作是判断这部分时间来自哪里。

诊断结果有两点。第一，intra-node 共享内存 MPI 使用 eager protocol，
\code{MPI\_Isend} 返回时数据通常已完成本地拷贝，\code{Waitall} 的主要等待来自
各 rank 到达等待点的时间差。第二，np$\ge 3$ 时上下边界 rank 与中间 rank 的 halo
工作量不同，\code{vert\_pack} 和 \code{vert\_unpack} 的 max/min 差异会直接放大
\code{waitall}。

本轮还做了两项否定实验。其一是交错 pack，在每个 $k$ 块后提前发起
\code{MPI\_Isend}；结果 FOM 下降约 3\%，非阻塞调用次数增加带来的开销更大。
其二是 split collision，将 border collision 提前到 \code{halo\_start} 之前；
结果 FOM 下降 34\%，额外的 OMP 区域触发了明显的 worker 唤醒/休眠成本。
opt25 的结论很直接：问题在 overlap 窗口和结构不均衡，继续细抠当前通信提交顺序收益有限。

\subsubsection{单一 \texttt{omp parallel} 区域重构——opt26}

opt26 处理的是 opt25 中 split collision 的结构问题。改动目标是保留更长的
overlap 窗口，同时避免重复进入 OMP 并行区。主循环在
\code{has\_vn=true} 路径下重写为一个外层 \code{\#pragma omp parallel}，
内部顺序为 border collision、\code{halo\_start}、interior collision、
\code{propagation\_interior}、\code{halo\_finish}、\code{propagation\_border}；
并行工作使用 \code{omp for}，MPI 调用限制在 \code{omp master} 中。

为此新增三组接口，均要求调用方已处于 \code{omp parallel} 区域内：
\code{collision\_rows}（按行范围，含隐式 barrier）、
\code{propagation\_interior\_omp\_region}（内部用 \code{nowait}，末尾显式 barrier）、
\code{propagation\_border\_omp\_region}（同）。
主循环在 \code{has\_vn} 路径下的结构如下：

\begin{lstlisting}[language={}]
omp parallel:
    collision_rows(border: j=1, j=H-2)      // → barrier
    omp master: halo_start()                 // 无 barrier（interior 不读 ghost row）
    collision_rows(interior: j=2..H-3)       // 与 MPI 传输重叠 → barrier
    propagation_interior()                   // 与 MPI 传输重叠 → barrier
    omp master: halo_finish()               // 写入 ghost row
    omp barrier                             // ghost row 可见后再继续
    propagation_border()                    // → barrier
\end{lstlisting}

barrier 只保留在依赖关系需要的位置：border collision 结束后再允许
master 打包；\code{halo\_start} 与 interior collision 之间不加 barrier；
\code{halo\_finish} 之后保留一次显式 barrier，保证 ghost row 写入完成后再进入
\code{propagation\_border}。
