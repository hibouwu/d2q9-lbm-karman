# 优化25：waitall 归因分析与诊断

## 1. 背景

优化24 profiling 显示（np=4）：`waitall` 占 loop total 的 25%，avg=0.878s，max=1.015s，是最大单一瓶颈。本轮工作以**归因分析为优先目标**，不急于改通信逻辑，而是先弄清楚 `waitall` 在等什么，再评估是否有可实施的针对性优化。

**本轮实际工作**：
1. 审阅通信结构与 profiling 设计，提出根因假设
2. 增加 `min_total_s` 到 profiling report（诊断性修改）
3. 运行诊断基准，获取 min/avg/max 三值分布
4. 评估"interleaved pack+Isend"优化，实验后发现无效并撤回
5. 完成归因结论

---

## 2. 审阅结论：waitall 等的是什么

### 2.1 waitall 的 request 组成

`MPI_Waitall` 等的是 `halo_exchange_start` 中发起的最多 4 个 async requests：

- `MPI_Irecv` from top_id（若有邻居）
- `MPI_Irecv` from bottom_id（若有邻居）
- `MPI_Isend` to bottom_id（若有邻居）
- `MPI_Isend` to top_id（若有邻居）

**完全是 vertical halo 的 send/recv**，无其他 request 混入。horizontal 和 diagonal 以阻塞方式独立运行，不在 Waitall 范围内。

### 2.2 overlap 结构

当前 main loop 的 overlap 设计：

```
special_cells → collision → halo_start(pack+Isend) → propagation_interior → Waitall+unpack → propagation_border
```

overlap 窗口 = `propagation_interior` 耗时。

| 配置 | prop_interior/call | waitall/call | 是否足够 |
|---|---|---|---|
| np=2 OMP=4 | 0.034ms | 0.016ms | ✓ prop > waitall |
| np=4 OMP=2 | 0.037ms | 0.043ms | **✗ waitall > prop** |

np=4 下 overlap 已经不够。

### 2.3 边界 rank 结构性不均衡

np=4 的分割：edge rank（0/3）只有 1 个 vertical 邻居，middle rank（1/2）有 2 个。因此：
- edge rank：pack 1 行，更早发出 Isend
- middle rank：pack 2 行，Isend 晚发出约 1 个 pack 时间

Edge rank 更早进入 Waitall，等待 middle rank 的数据到达，承担了更多 waitall 时间。

### 2.4 当前 profiling 缺少 min，无法量化不均衡范围

现有 avg_total_s 和 max_total_s，缺 min_total_s。加 min 只需一次 `MPI_Reduce(MPI_MIN)`，是诊断 rank 间不均衡的最直接方法。

---

## 3. 诊断修改：新增 min_total_s

在 `src/lbm/profiling.cpp` 中增加第三次 `MPI_Reduce(MPI_MIN)`，profiling 表头新增 `min_total_s` 列：

```cpp
double min_t[PROF_COUNT];
MPI_Reduce(s_total, min_t, PROF_COUNT, MPI_DOUBLE, MPI_MIN, 0, MPI_COMM_WORLD);
```

报告格式变更：
```
Phase          avg_total_s  max_total_s  min_total_s  avg/call  pct_loop
```

Notes 补充：`min_total_s : fastest rank's total — imbalance = max - min`

## 4. 性能测试配置

**测试口径**（与优化24一致，禁止超线程干扰）：

```bash
# np=2
OMP_NUM_THREADS=4 mpirun -np 2 build/top.lbm-exe bench_config.txt

# np=4
OMP_NUM_THREADS=2 mpirun -np 4 build/top.lbm-exe bench_config.txt
```

`OMP_NUM_THREADS` 按进程数等分物理核数（机器 8 核），避免超额认购。np>1 场景下不加 `OMP_PROC_BIND=close` 强制绑定（单 NUMA 节点下会造成进程间核争用，见优化24记录）。

## 5. 诊断结果：min_total_s 数据

### 5.1 np=2 诊断（优化25 基准，代码未变）

| Phase | avg | max | min | max/min |
|---|---:|---:|---:|---:|
| collision | 0.815s | 0.881s | 0.749s | 1.18 |
| propagation_interior | 0.683s | 0.713s | 0.654s | 1.09 |
| **halo: waitall** | **0.324s** | **0.444s** | **0.205s** | **2.17** |
| halo: vert_pack | 0.163s | 0.169s | 0.158s | 1.07 |
| halo: vert_unpack | 0.161s | 0.165s | 0.157s | 1.05 |
| FOM | — | — | — | **860.97 MLUPS** |

**关键发现**：
- waitall **min = 0.205s（非零）**：即便"快"的 rank 也等了显著时间，说明不是单向等待
- waitall **max/min = 2.17×**：两个 rank 的等待时间差 2 倍，存在明显不均衡
- collision 和 propagation_interior 的 max/min 仅 1.09–1.18，计算负载基本均衡

### 5.2 np=4 诊断（优化25 基准，代码未变）

| Phase | avg | max | min | max/min |
|---|---:|---:|---:|---:|

| propagation_interior | 0.741s | 0.818s | 0.699s | 1.17 |

**关键发现**：

- waitall min = 0.537s（非零），max/min = 2.24×：edge rank 始终等待更多
- vert_unpack 也有 1.71× 不均衡，原因相同（middle rank 接收 2 行，unpack 更多）

## 6. waitall 根因归因

### 6.1 确认：结构性不均衡（主因，np≥3）

vert_pack max/min = 1.92× 直接验证了 edge vs middle rank 的结构差异：
- middle rank 打包 2 行，pack 约 2× 于 edge rank，Isend 延迟发出
- edge rank 先完成 pack+Isend，进入 propagation_interior 后更早到达 Waitall
- edge rank 在 Waitall 里等待 middle rank 的数据，等待时间更长

这是 **程序结构固有的不均衡**，只要 np≥3 且使用均匀 1D 垂直分割，就必然存在 edge/middle 差异。

### 6.2 确认：OMP 调度抖动（次因，所有 np）

np=2 两个 rank 结构完全对称（都是 edge rank），理论上计算时间相同。但 waitall min = 0.205s（非零），且 max/min = 2.17×，说明：
- 每一步中，哪个 rank 先完成 propagation_interior 是随机的（OMP parallel for 最慢线程决定完成时间）
- 先完成的 rank 进入 Waitall 等对方；后完成的 rank 到达时消息已就绪，等待时间短
- 20000 步累积下来，两个 rank 各自承担约一半的"先到"角色，所以 min ≠ 0

### 6.3 否定：网络传输延迟

intra-node shared-memory MPI 的消息传输时间 < 0.001ms（57 KB / 20 GB/s ≈ 0.003ms，实测 Isend 到完成约 0.001ms）。waitall/call = 0.016ms（np=2），比传输时间大 16 倍，说明主要是等待对方 rank 进入 Waitall，而非消息本身延迟。

## 7. interleaved pack+Isend 实验（尝试后撤回）

### 7.1 假设

对于 np=4 middle rank，当前代码先打包 sbuf_bot、再打包 sbuf_top、再统一发送。改为打包 sbuf_bot 后立即发送（sbuf_bot 在传输中），再打包 sbuf_top 后发送，可让 sbuf_bot 更早到达对方，缩短 edge rank 的等待。

理论收益：sbuf_bot 早发出约 1 个 pack_per_row 时间（≈ 0.006ms），对 edge rank 的 waitall/call 有约 10% 改善。

### 7.2 实验结果（np=4）

| Phase | 优化25 诊断（原始） | interleaved（撤回） | Δ |
|---|---:|---:|---:|
| FOM | 716.32 MLUPS | 682.41 MLUPS | **−4.7%** |
| waitall avg | 0.852s | 1.043s | **+22%** |
| waitall max | 1.204s | 1.531s | +27% |
| waitall min | 0.537s | 0.363s | −32% |
| vert_pack | 0.204s | 0.213s | +4% |
| vert_post | 0.011s | 0.004s | −66% |

实验结果为**负收益**。interleaved 版本 waitall 反而更差。

### 7.3 原因分析

intra-node MPI（shared memory）下，Isend 调用本身即完成数据拷贝（eager protocol），发出时间点差异（0.006ms）对接收方 Waitall 的可见时间影响极小。更重要的是，interleaved 改变了 Irecv/Isend 的提交顺序，可能影响 OpenMPI 的 eager buffer 匹配和进度推进时序，导致轻微的额外开销。

### 7.4 结论

interleaved pack+Isend 优化已撤回，代码恢复 pack-all-then-send 结构。min_total_s profiling 保留。

## 9. 结论

| 问题 | 结论 |
|---|---|
| waitall 主要在等什么 | **对方 rank 进入 Waitall / 发出 Isend 的时间差**，不是网络延迟 |
| 根因一 | edge vs middle rank 结构性不均衡（np≥3），vert_pack max/min = 1.92× 直接证实 |
| 根因二 | OMP parallel for 的步间调度抖动，np=2 min≠0 证实 |
| interleaved 优化 | 实验结果为负收益，已撤回 |
| 是否值得继续做通信层优化 | **当前 intra-node 通信延迟不是瓶颈**，真正的问题是 overlap 不够 + 结构不均衡 |

## 10. collision_border/interior split 实验（尝试后撤回）

### 10.1 假设

将 collision 分为两阶段：collision_border（j=1, j=H-2，串行）→ halo_start → collision_interior（j=2..H-3，OMP 并行）。理论 overlap 窗口 = collision_interior + propagation_interior ≈ 0.074ms >> waitall ≈ 0.043ms。

### 10.2 实现尝试

- 在 `physics.cpp` 增加 `collision_border`（无 OMP，仅 800 列 × 2 行串行循环）和 `collision_interior`（调 `collision_multicell_vec_range`，`#pragma omp parallel for`）
- 在 `main.cpp` 重构 has_vn=true 路径：`collision_border → halo_start → collision_interior → propagation_interior → halo_finish → propagation_border`

### 10.3 实验结果（np=2）

| Phase | 优化25 诊断 | split 实验（撤回） | Δ |
| --- | ---: | ---: | ---: |
| FOM | 860.97 MLUPS | **566.70 MLUPS** | **−34%** |
| collision | 0.815s | 1.775s | **+118%** |
| halo: waitall | 0.324s | 0.815s | **+152%** |

### 10.4 根因分析

实验结果为**严重负收益**，原因是 OMP 线程休眠唤醒开销：

- `collision_border` 是纯串行代码（无 `#pragma omp parallel`）。OMP worker 线程在此期间脱离活跃状态，可能进入休眠（取决于 `GOMP_SPINCOUNT`/`OMP_WAIT_POLICY` 默认配置）
- `collision_interior` 调用 `#pragma omp parallel for` 时，运行时需要重新唤醒休眠的 worker 线程，代价约 **~100μs 每次**
- 20000 步 × 100μs = **2000ms 额外唤醒开销**，占比超过 collision 总时间，collision 从 0.815s 暴涨至 1.775s

本质原因：当前代码结构以"顶层函数调用串行流"为基础，OMP 并行全部通过 `#pragma omp parallel for` 在各函数内部按需创建，函数之间隐含线程同步。将 collision 拆为串行+并行两段，破坏了 OMP 线程的连续活跃性，导致不可接受的唤醒开销。

### 10.5 正确的延长 overlap 方案（需要较大重构）

要正确实现 collision_border/interior split，需要一个**跨越整个 border→halo_start→interior 序列的单一 `#pragma omp parallel` 区域**：

```cpp
#pragma omp parallel
{
  // 并行 border collision（#pragma omp for，隐式 barrier）
  // master 线程串行执行 halo_start（#pragma omp master + barrier）
  // 并行 interior collision（#pragma omp for）
  // 并行 propagation_interior（#pragma omp for）
}
```

这需要：(1) halo_start 对 OMP master 安全，(2) 将 profiling 宏改为 master-only，(3) 整个计算核心移入一个 parallel 区域。工程量较大，留作后续优化。

### 10.6 结论

collision_border/interior split 已完全撤回，代码恢复到优化25基准（仅保留 min_total_s profiling）。

## 11. 后续工作

### 最高优先级：延长 overlap 窗口（需要 OMP 单一 parallel 区域重构）

理论上可行，但需要将 main loop 的 border→halo→interior 序列包在一个 `#pragma omp parallel` 区域内，并使 halo_start 对 OMP master-only 安全。这是一次较大的架构性重构。

### 次优先级：结构性不均衡（np=4 edge/middle rank）

可通过非均匀分割（给 middle rank 分配稍少行数以补偿其更多的 halo 工作）来均衡负载，但需要改动 mesh 分割逻辑，复杂度较高。

### 可选：MPI persistent requests

用 `MPI_Send_init` + `MPI_Start` 替代每步的 `MPI_Isend`，减少通信初始化开销（当前 vert_post ≈ 0.011ms/call，有一定节省空间）。
