# 优化20 — SoA k-outer propagation

**日期：** 2026-04-27
**分类：** P3 / SoA kernel 重组
**状态：** 完成；正式 benchmark 通过

---

## 0. 背景与目标

优化19完成了 AoS → SoA 重构，并通过代码审查形成了 **SoA baseline v1.1**，
但当时的 benchmark 已经揭示：SoA baseline 比 AoS 慢，主要原因不是数值实现
问题，而是 propagation kernel 的结构性访存问题。

优化20的目标：

1. 将 propagation 改成真正面向 SoA 的 **k-outer** 实现，消除 9-plane 跨平面 gather
2. 以 **SoA baseline v1.1 vs SoA optimized** 为正式比较口径
3. 附加实验：同时测试 unrolled collision，观察组合收益

**不与 AoS baseline 直接比较**，那已经是优化19的结论。

正式基线：SoA baseline v1.1（优化19），以下简称"SoA v1.1"

## 1. 为什么 SoA v1.1 的 propagation 慢

SoA v1.1 使用 per-cell k-inner 结构：

```cpp
// per-cell baseline（SoA v1.1）
for i, j:
    for k in 0..8:
        out[k][i][j] = in[k][i - dx[k]][j - dy[k]]
```

对于每个 cell (i,j)，内层 k 循环访问 9 个不同的方向 plane：

- 每个 plane 大小 = W×H×8 = 802×162×8 ≈ 1 MB
- 每次 `propagate_cell` 从 9 个相距 ≈1MB 的位置读，写到 9 个同样分散的位置
- 单次 `propagation_interior` 工作集 = 9 planes × 1MB = **9MB**，超出 L2/L3
- 对比 AoS：一列 9×162×8 ≈ 11KB，9 个邻居列 ≈ 100KB，完全在 L2 内

这是 SoA "反模式"：用了 SoA 存储却仍按 AoS 心智模型逐 cell 访问。

---

## 2. k-outer 为什么有利

对每个方向 k（偏移 (dx, dy)），k-outer 写成：

```cpp
for k in 0..8:
    plane_out = Mesh_get_plane(out, k)
    plane_in  = Mesh_get_plane(in, k)
    for i in [i_lo, i_hi):
        // 对固定 i，src = plane_in[(i-dx)*H - dy], dst = plane_out + i*H
        // j 循环：stride-1 连续读写 → memcpy
        memcpy(dst + j_lo, src + j_lo, (j_hi - j_lo) * 8)
```

关键性质：

- 固定 k：只访问 1 对 plane（读 in[k]，写 out[k]），工作集 **≈2MB**
- 内层 j 循环：stride-1 连续访问，可直接用 `memcpy` 或 SIMD store/load
- 9 个 k 迭代串行，每次都完全在 L3 cache 内
- 可加 `#pragma omp simd` 或 `memcpy`（libc 自动选最宽 SIMD）

## 3. 实现改动

### 3.1 核心变更

**文件：** `src/lbm/physics.cpp`

删除 `propagate_cell` 辅助函数，重写三个 propagation 函数为 k-outer 结构。

**索引范围计算（以 direction k 的 dx, dy 为例）：**

```cpp
const int i_lo = (dx > 0) ? dx : 0;   // source i-dx must stay in [0,W)
const int i_hi = (dx < 0) ? W + dx : W;
const int j_lo = (dy > 0) ? dy : 0;   // source j-dy must stay in [0,H)
const int j_hi = (dy < 0) ? H + dy : H;
```

对于 k=0（dx=dy=0）：全范围 [0,W)×[0,H)，memcpy 整列。

**列级 memcpy：**

```cpp
// row_in[j] = plane_in[(i-dx)*H + (j-dy)]  →  pointer: plane_in+(i-dx)*H - dy
const double* bi = pin + (i - dx) * H - dy;
double*       bo = pout + i * H;
memcpy(bo + j_lo, bi + j_lo, (j_hi - j_lo) * sizeof(double));
```

读写均为 stride-1（j 方向），`memcpy` 使用 SIMD 宽指令（典型 32/64 字节/周期）。

### 3.2 propagation_interior overlap 语义保留

`propagation_interior(has_vert_neighbors=true)` 跳过 j=1 和 j=H-2：

```cpp
// Interior: j=0, j=2..H-3, j=H-1 (skip j=1, j=H-2 pending ghost rows)
const int jm_lo = max(2, j_lo);
const int jm_hi = min(H-2, j_hi);

for i in [i_lo, i_hi):
    if j_lo == 0: bo[0] = bi[0]          // top ghost output (stale src ok)
    memcpy(bo + jm_lo, bi + jm_lo, ...)   // j=2..H-3
    if j_hi == H: bo[H-1] = bi[H-1]      // bottom ghost output
```

`propagation_border` 在 `halo_finish` 后处理 j=1 和 j=H-2：

```cpp
// For dy∈{-1,0,1}: sj=1-dy∈{0,1,2} and sj=(H-2)-dy∈{H-3,H-2,H-1} — always valid
bo[1]     = bi[1];
bo[H - 2] = bi[H - 2];
```

main.cpp 的 overlap 调用顺序不变：

```text
halo_exchange_start()     → 发送 border rows，同时开始 interior propagation
propagation_interior()    → 处理 j=0, 2..H-3, H-1（不读 ghost rows）
halo_exchange_finish()    → Waitall + unpack ghost rows
propagation_border()      → 处理 j=1, j=H-2（读 freshly-received ghost rows）
```

### 3.3 OpenMP 结构

```cpp
#pragma omp parallel          // 线程团只创建一次
{
    for (int k = 0; k < DIRECTIONS; k++) {   // 所有线程串行遍历 k
        ...
        #pragma omp for schedule(static) nowait   // 并行分配 i 迭代
        for (int i = i_lo; i < i_hi; i++) { memcpy(...); }
        // nowait: 快线程立刻进入下一个 k（不同平面无冲突）
    }
    // 隐式 barrier 在 omp parallel 结束处
}
```

线程团创建开销 1 次而非 9 次；`nowait` 消除 k 间不必要的 barrier（不同 k 访问不同 plane，无数据依赖）。

### 3.4 halo pack/unpack

本轮**不改**格式，继续使用 cell-interleaved（cell-major）。

原因：

- 修改 pack 格式需双端同步（send 和 recv 必须一致）
- 当前测试拓扑（np=1/2/4, width=800）无水平邻居，垂直 pack 的优化有收益但不是主要瓶颈
- pack/unpack 优化留优化21

### 3.5 collision

本轮不修改 collision kernel，通过 `LBM_COLLISION_IMPL` 环境变量提供两条路径：

- `baseline`（默认）：2-pass loop 融合，用于隔离 propagation 改造的收益
- `unrolled`：手工展开 BGK，用于测量组合收益

## 5. 分析

### 5.1 优化20主要收益来自 propagation

从 SoA v1.1 到 SoA k-outer（baseline collision）：

- FOM 提升 +37%～+48%，完全来自 propagation 从 per-cell 9-plane gather 改为
  k-outer stride-1 memcpy
- collision 时间基本不变（-1% 到 +3%，误差范围内）

### 5.2 k-outer 收益来源

- **工作集从 9MB 降至 2MB**：每个 k 迭代只访问 1 对 plane（读 + 写），完整
  fitting 入 L3
- **memcpy 效率**：stride-1 写 + 连续读，libc memcpy 可用 32/64 字节 SIMD 宽操作，
  接近内存带宽上限
- **OMP 线程复用**：一个 `#pragma omp parallel` 块覆盖所有 9 个 k 方向，
  线程 fork/join 开销降至 1 次；`nowait` 进一步允许快线程提前进入下一平面

### 5.3 np=4 收益较 np=1/2 略小

np=4 的瓶颈从 propagation 转移到 halo 通信：

```text
np=4, SoA k-outer + baseline:
  collision = 49.54%
  halo_finish = 30.25%  (waitall=20.29% + vert_pack/unpack=15.5%)
  propagation_interior = 6.47%
```

propagation 已基本消除，剩余回退来自 vertical halo 的 pack/unpack：
`vert_pack + vert_unpack = 0.40s + 0.71s = 1.11s = 15.5%`

这是下一个优化目标（优化21）。

### 5.4 unrolled collision 的收益分析

unrolled 比 baseline collision 快 24%～36%，原因：

- 9 次标量 load 的顺序已消除循环 overhead
- `f0..f8` 作为独立局部变量，编译器可做更激进的寄存器分配 + ILP
- 相比 baseline 的两个 k 循环（各自 gather），unrolled 只做一组 9 次 load + 一组
  9 次 store，内存访问次数相同但指令序列更适合乱序执行

注意：np=4 时 unrolled collision 导致 halo_finish 占比上升至 37%（=collision
占比），说明 collision 被显著加速后 halo 等待成为主要瓶颈，而非 collision 本身。

## 7. 瓶颈转移与优化21方向

经过优化20，各配置主要瓶颈：

| 配置 | 当前主要瓶颈 | 次要瓶颈 |
| --- | --- | --- |
| `np=1, OMP=4` | collision（83%） | — |
| `np=2, OMP=4` | collision（68%）+ halo_finish（13%） | propagation_border（2.5%） |
| `np=4, OMP=4` | collision（50%）+ halo_finish（30%） | vert_pack/unpack（16%） |

**优化21建议（按优先级）：**

1. **halo vertical pack/unpack 优化**（np=4 收益最明显）
   - 当前 cell-interleaved pack：gather from 9 planes per cell，stride-H 访问
   - 改为 k-major format：每个 k 连续扫描一行，stride-H 仍然，但可 SIMD 化或用
     `memcpy`；注意双端格式必须同步

2. **collision 进一步优化**（np=1/2 主要瓶颈）
   - unrolled collision 已有基础，可考虑针对 SoA plane pointer 的优化
   - 外层 j 循环（而非外层 i）可能更有利于 SIMD：固定 i，遍历 j 的列是 stride-1 的
   - 但 collision 需要 9 plane 的 cross-gather，很难完全消除 gather 开销

3. **propagation_border OMP 负载**
   - propagation_border 每步处理 2 行（j=1, j=H-2），OMP 线程分摊 W 次迭代
   - 对于 W=802, 4 线程：每线程约 200 列，开销已较小；如有需要可考虑 non-parallel
