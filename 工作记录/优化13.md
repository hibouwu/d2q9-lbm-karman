## 1. 问题描述

P2-2 将 `propagation` 改为 pull 模式后，每个输出格点 `(i,j)` 的所有 9 个方向值只由该格点自己的循环写入，**完全无写竞争**。

这是 OpenMP 并行化的充分条件：外层 i 循环按列分配给不同线程，每个线程写自己负责的列，各列数据在内存中不重叠（步幅 `height * DIRECTIONS * 8 = 11520` 字节），cache 行不共享。

### 1.1 对比 push 模式为何不能并行

push 模式：线程 A 处理源格点 `(i,j,k)`，线程 B 处理源格点 `(i',j',k')`，如果两者计算出相同目标 `(ii,jj)`，则发生写竞争。例如方向 {+1,0} 和方向 {-1,0} 的两个不同源格点同时写同一目标行，即为典型竞争场景。

### 1.2 pull 模式并行安全性

- 目标格点 `(i,j)` 的写入完全由外层 i 的分配决定
- 同一线程内，j 循环顺序写 `mesh_out[i][0..height-1]`（连续内存）
- 源格点读取来自 `mesh_in[i-dx][j-dy]`，方向偏移最大 1 格，只会跨越相邻列，读取只读不写，无竞争

## 2. 修改内容

**文件：** `src/lbm/physics.cpp` → `propagation`

```diff
+#pragma omp parallel for schedule(static)
 for (size_t i = 0; i < mesh_out->width; i++) {
   for (size_t j = 0; j < mesh_out->height; j++) {
     for (size_t k = 0; k < DIRECTIONS; k++) {
       ssize_t si = (ssize_t)i - (ssize_t)direction_matrix[k][0];
       ssize_t sj = (ssize_t)j - (ssize_t)direction_matrix[k][1];
       if (si >= 0 && si < (ssize_t)mesh_in->width && sj >= 0 && sj < (ssize_t)mesh_in->height) {
         Mesh_get_cell(mesh_out, i, j)[k] = Mesh_get_cell(mesh_in, si, sj)[k];
       }
     }
   }
 }
```

## 5. 结论

- pull 模式使 `propagation` 可安全并行化，4 线程提升 67%
- `propagation` 是 memory-bandwidth-bound 的热点（每格点 9 次读 + 9 次写），线程扩展性好
- 结合 P2-3（collision OMP）和 P2-5（propagation OMP），4 线程下 np=1 从 60 MLUPS 提升至 ~194 MLUPS（+220%）
- MPI+OpenMP 混合：np=4, OMP=4 达 ~202 MLUPS，在单台机器上充分利用所有硬件并行度
