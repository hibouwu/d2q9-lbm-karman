# 优化03 — 修复垂直通信收发大小不匹配（P0）

**日期：** 2026-04-19  
**分类：** P0 / 正确性保障 + 通信健壮性  
**状态：** 完成，正确性已验证

---

## 1. 问题描述

### 1.1 发现位置

`src/lbm/communications.cpp`，`lbm_comm_sync_ghosts_vertical` 函数的 RECV 分支：

```cpp
case COMM_SEND:
  for (size_t x = 1; x < mesh_to_process->width - 2; x++) {
    for (size_t k = 0; k < DIRECTIONS; k++) {
      MPI_Send(&Mesh_get_cell(mesh_to_process, x, y)[k],
               1,           // ← 发送 1 个 double
               MPI_DOUBLE, target_rank, k, MPI_COMM_WORLD);
    }
  }

case COMM_RECV:
  for (size_t x = 1; x < mesh_to_process->width - 2; x++) {
    for (size_t k = 0; k < DIRECTIONS; k++) {
      MPI_Recv(&Mesh_get_cell(mesh_to_process, x, y)[k],
               DIRECTIONS,  // ← 接收 9 个 double！与发送不一致
               MPI_DOUBLE, target_rank, k, MPI_COMM_WORLD, &status);
    }
  }
```

### 1.2 问题分析

**收发大小不匹配：**
- `MPI_Send` count = 1（发送 1 个 `double`，8 字节）
- `MPI_Recv` count = `DIRECTIONS` = 9（缓冲区声明接收 9 个 `double`，72 字节）

**MPI 标准行为：**  
当接收缓冲区大于实际消息时，MPI 不报错，只填充实际收到的 1 个元素，其余 8 个元素不被覆盖（保留旧值）。虽然不会立即崩溃，但这是一个设计缺陷：

1. **可移植性风险：** 某些 MPI 实现（或严格模式）会将此视为错误
2. **调试混淆：** 收发大小不一致会导致性能分析工具报告警告
3. **未来改造障碍：** P1 阶段将重构为批量通信；当前不一致会使重构容易出错

**性能维度（虽然不影响当前正确性）：**  
垂直方向通信的极细粒度本身就是性能瓶颈：对一行 `width-2 = 798` 个格点，每个格点 9 个方向，产生 **7182 条独立 MPI 消息**（每次 halo exchange 仅垂直一个方向就有这么多）。这将在 P1 阶段通过批量通信彻底重构。

---

## 2. 修改内容

**文件：** `src/lbm/communications.cpp`，`lbm_comm_sync_ghosts_vertical` RECV 分支

```diff
 MPI_Recv(
   &Mesh_get_cell(mesh_to_process, x, y)[k],
-  DIRECTIONS,
+  1,
   MPI_DOUBLE,
   target_rank,
   k,
   MPI_COMM_WORLD,
   &status
 );
```

收发 count 统一为 1，与 SEND 端一致。

### 通信消息数分析

当前垂直方向通信开销（修复后仍然是细粒度，留待 P1 重构）：

| 方向 | 消息条数/halo exchange（np=2，width=800） |
|---|---|
| 垂直（每侧）| (800-2) × 9 = **7182 条** |
| 水平（每侧）| 80 行（每行 1 条）= **80 条** |

垂直方向远多于水平方向，P1 优化将把 7182 条消息合并为 1 条。

- 修复收发大小不一致，消除了潜在的可移植性风险和调试混淆
- 当前仍为细粒度逐元素通信，性能优化留待 P1 阶段（批量发送）
- 不影响物理计算结果，正确性验证通过
