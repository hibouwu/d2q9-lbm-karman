## 1. 问题描述

### 1.1 原始代码

```cpp
// lbm_comm_sync_ghosts_vertical — 原始版本（P0-3 修复后）
case COMM_SEND:
  for (size_t x = 1; x < mesh->width - 1; x++) {
    MPI_Send(Mesh_get_cell(mesh_to_process, x, y), 1, MPI_DOUBLE,
             target_rank, 0, MPI_COMM_WORLD);
  }
case COMM_RECV:
  for (size_t x = 1; x < mesh->width - 1; x++) {
    MPI_Recv(Mesh_get_cell(mesh_to_process, x, y), 1, MPI_DOUBLE,
             target_rank, 0, MPI_COMM_WORLD, &status);
  }
```

对一行 `width-2` 个格点，每个格点单独发一条消息（每条仅含 1 个 `MPI_DOUBLE`），产生 **`width-2 = 798` 条消息**。

### 1.2 内存布局分析

数组布局：`cells[(x * height + y) * DIRECTIONS]`（x 为外层，列主序）

行 y 的相邻格点 `(1,y), (2,y), ..., (width-2,y)` 在内存中**不连续**：  
相邻格点间距 = `height * DIRECTIONS * 8 = 82 * 9 * 8 = 5904` 字节。

因此不能直接整行发送，需要：
1. **发送前打包**：将行 y 的 `width-3` 个内部格点（x=1..width-3）复制到连续缓冲区
2. **一次 MPI_Send**：发送整个缓冲区，count = `(width-3) * DIRECTIONS`
3. **接收后解包**：将接收到的缓冲区内容复制回各格点

`mesh_comm->buffer` 在 `lbm_comm_init` 中已按需分配：
```cpp
mesh_comm->buffer = static_cast<double*>(malloc(sizeof(double) * DIRECTIONS * width / nb_x));
```
大小 = `DIRECTIONS * (width/nb_y)` doubles，恰好容纳一整行内部格点。

> 注：此处 `ncells = mesh_to_process->width - 3`，对应 x=1..width-3（不含两侧幽灵列）。

### 1.3 消息数对比

| 方向 | 原始（per 行，per side）| 优化后 |
|---|---|---|
| 垂直（1 行） | 798 条消息 | **1 条消息** |
| 垂直（上+下两侧）| 1596 条消息 | **2 条消息** |

函数体：

```cpp
static void lbm_comm_sync_ghosts_vertical(
  lbm_comm_t* mesh_comm,
  Mesh* mesh_to_process,
  lbm_comm_type_t comm_type,
  int target_rank,
  uint32_t y
) {
  if (target_rank == -1) {
    return;
  }

  const size_t ncells = mesh_to_process->width - 3; // x = 1 .. width-3 inclusive
  const size_t count  = ncells * DIRECTIONS;
  double* buf         = mesh_comm->buffer;
  MPI_Status status;

  switch (comm_type) {
  case COMM_SEND:
    for (size_t x = 1; x < mesh_to_process->width - 2; x++) {
      memcpy(buf + (x - 1) * DIRECTIONS, Mesh_get_cell(mesh_to_process, x, y), DIRECTIONS * sizeof(double));
    }
    MPI_Send(buf, count, MPI_DOUBLE, target_rank, 0, MPI_COMM_WORLD);
    break;
  case COMM_RECV:
    MPI_Recv(buf, count, MPI_DOUBLE, target_rank, 0, MPI_COMM_WORLD, &status);
    for (size_t x = 1; x < mesh_to_process->width - 2; x++) {
      memcpy(Mesh_get_cell(mesh_to_process, x, y), buf + (x - 1) * DIRECTIONS, DIRECTIONS * sizeof(double));
    }
    break;
  default:
    fatal("unknown type of communication");
  }
}
```

## 3. 性能数据（2000 步，无输出，3 次平均）

| 进程数 | P1-4 后 | P1-5 后 | 变化 |
|---|---|---|---|
| np=1 | 59.01 MLUPS | 58.57 MLUPS | 持平（单进程无通信） |
| np=2 | 63.23 MLUPS | 102.75 MLUPS | **+62%** |
| np=4 | 33.27 MLUPS | 76.73 MLUPS | **+131%** |

np=2/4 大幅提升的原因：np=2 时 nb_y=2，每次 halo_exchange 实际触发 2 条垂直通信（top/bottom 各 1 条），将每条 798 消息压缩为 1 条，消息数减少 798×。

### P1 全阶段汇总（对比 P0 基线）

| 进程数 | P0 基线 | P1 全部完成后 | 总提升 |
|---|---|---|---|
| np=1 | 60.16 MLUPS | 58.57 MLUPS | 持平 |
| np=2 | 68.66 MLUPS | 102.75 MLUPS | **+50%** |
| np=4 | 28.84 MLUPS | 76.73 MLUPS | **+166%** |

> P0 基线已移除隐藏 sleep，为真实通信性能。

## 5. 结论

- 垂直通信从 `width-2 = 798` 条单格点消息缩减为 1 条打包消息
- 打包/解包通过 `memcpy` 完成，访问 `mesh_comm->buffer`（已在初始化时分配）
- np=2 提升 62%，np=4 提升 131%，是 P1 阶段收益最大的单项优化
- 垂直通信是主要通信路径（对 2 的幂次进程数 nb_y=np），水平对角通信 target=-1 均 skip
