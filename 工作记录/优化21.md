# 优化21 — SoA multi-cell collision

**日期：** 2026-04-27
**分类：** P2 / collision kernel 重组
**状态：** 完成；正式 benchmark 通过

---

## 0. 背景与目标

优化20 完成了 k-outer propagation，瓶颈从 propagation 转移至 collision：

```text
np=1, SoA k-outer + unrolled:
  collision = 78.99%（7.67s）
  propagation_interior = 10.67%
np=4, SoA k-outer + unrolled:
  collision = 38.39%（2.24s）
  halo_finish = 34.26%（MPI wait）
```

优化20 附加实验已经测试了 SoA `unrolled` collision（per-cell 手工展开）。
优化21 的目标是做真正的 **SoA multi-cell 优化**：不是进一步展开单 cell，
而是改变 kernel 结构，让 SIMD lane 跨多个 cell（j 方向），而不是多个方向。

### 0.1 与优化20正式口径的关系

需要明确区分两条比较链：

1. **优化20正式主线**
   - 比较对象：`SoA baseline v1.1` vs `SoA k-outer + baseline`
   - 目的：隔离并验证 propagation 从 per-cell SoA 改为 k-outer SoA 的收益

2. **优化21正式主线**
   - 比较对象：`SoA k-outer + unrolled` vs `SoA k-outer + multicell`
   - 目的：在 propagation 已固定为 k-outer 的前提下，继续优化 collision kernel

因此，优化21并不是绕过优化20的正式结论，而是**沿着优化20的附加实验路径继续推进 collision**：

- 优化20先证明：`k-outer propagation` 本身有效
- 优化20附加实验再证明：`unrolled collision` 在该基础上有额外收益
- 优化21继续证明：`multicell collision` 又优于 `unrolled collision`

换言之，优化21的正式结论应理解为：

```text
SoA baseline v1.1
  →（优化20正式主线）k-outer + baseline
  →（优化20附加实验）k-outer + unrolled
  →（优化21正式主线）k-outer + multicell
```

后续若需要回顾完整收益链，应将以上四个阶段串联解读，而不是把优化21单独理解为“直接相对 SoA baseline v1.1 的唯一正式比较”。

## 1. 审查阶段：五个问题的回答

### Q1. 当前 collision 的主要瓶颈是什么？

当前 `unrolled` 路径：

```cpp
// 外层 i 循环（OMP 并行）
for (size_t i = 1; i < W - 1; i++) {
    for (size_t j = 1; j < H - 1; j++) {
        compute_cell_collision_unrolled(mesh_out, mesh_in, i, j);
        // 内部：Mesh_get_f(in, k, x, y) = cells[k*W*H + x*H + y]
        // 对 j 内层：k*W*H + x*H + y 的 y 增量是 stride-1
        // 但函数调用边界使编译器无法识别 j 方向的 stride-1 模式
    }
}
```

**根本问题**：per-cell 调用约定隐藏了 j 方向的 stride-1 访问模式。
编译器看到 `Mesh_get_f(in, k, x, y)` 即使内联后也难以推断连续
j 值对每个 plane 都是 stride-1 → j 循环无法向量化 → 纯标量执行。

### Q2. SoA 单-cell unrolled 和 SoA multi-cell 的本质区别？

- **单-cell unrolled**：每次调用处理一个 cell，显式列出 f0..f8 为局部变量；
  SIMD lane 消耗在标量操作或浪费 → 无跨 cell 并行

- **multi-cell**：hoist 18 个 plane pointer（9 读 + 9 写），j 内层直接访问
  `cp_k[j]`（stride-1）；SIMD lane 对应不同 j 值（不同 cell）→
  编译器可用宽向量寄存器同时处理 4（AVX2）或 8（AVX-512）个 cell

关键区别：multi-cell 让向量化发生在 **cell** 维度（j），而非方向维度（k）。

### Q3. 最值得改的层次是哪一层？

**plane pointer hoist + j-stride-1 + `__restrict__`**，优先于：

- 手工 AVX-512 intrinsics（风险高，收益不确定，维护代价大）
- `#pragma omp simd`（会强制 `vdivpd zmm`，Zen 4 上 8 double 吞吐量 ≈ 1/24
  CPI，不如 8 个独立标量 `divsd` 流水线化的 ≈ 2 CPI）

在不增加显式 SIMD pragma 的情况下，编译器拿到 stride-1 + `__restrict__`
就能自动选择合适向量宽度，同时避免强制使用低吞吐量的向量除法。

### Q4. 本轮是否同时动 halo_finish？

**不动**。np=1 halo 占比 <0.05%，np=2 约 14%，np=4 约 34%。
先修 collision，再看改完后的 halo 相对占比，再决定优先级。

### Q5. LBM_COLLISION_IMPL 机制是否保留？

**保留并扩展**：新增 `multicell` 选项，并将其设为默认值。
`baseline` 和 `unrolled` 保留作为对照实验路径。

## 2. 实现改动

**文件：** `src/lbm/physics.cpp`

### 2.1 enum 扩展

```cpp
typedef enum {
  COLLISION_BASELINE,
  COLLISION_UNROLLED,
  COLLISION_MULTICELL,   // 新增
} lbm_collision_impl_t;
```

默认值从 `baseline` 改为 `multicell`：

```cpp
if (env == NULL || strcmp(env, "multicell") == 0) return COLLISION_MULTICELL;
```

### 2.2 collision_multicell 实现

核心思路：

```cpp
static void collision_multicell(Mesh* mesh_out, const Mesh* mesh_in) {
  // 常量提升至所有循环外部
  const double A      = 1.0 - RELAX_PARAMETER;
  const double rp_w0  = (4.0 / 9.0)  * RELAX_PARAMETER;
  const double rp_w14 = (1.0 / 9.0)  * RELAX_PARAMETER;
  const double rp_w58 = (1.0 / 36.0) * RELAX_PARAMETER;

  // 18 个 plane pointer（9 读 + 9 写），每个平面 W*H doubles，互不重叠
  const double* __restrict__ p0 = Mesh_get_plane(mesh_in,  0);
  // ... p1..p8
  double* __restrict__       q0 = Mesh_get_plane(mesh_out, 0);
  // ... q1..q8

#pragma omp parallel for schedule(static)
  for (int i = 1; i < W - 1; i++) {
    const size_t col_off = (size_t)i * H;
    // 列指针：固定 i，j 方向 stride-1
    const double* cp0 = p0 + col_off; /* ...cp1..cp8 */ ;
    double*       cq0 = q0 + col_off; /* ...cq1..cq8 */ ;

    for (int j = 1; j < H - 1; j++) {
      // 9 个 stride-1 load（来自 9 个独立 plane）
      const double f0 = cp0[j], f1 = cp1[j], ..., f8 = cp8[j];
      // BGK collision（同 unrolled，但 w0/w14/w58 换为 rp_wX * density）
      ...
      // 9 个 stride-1 store
      cq0[j] = ...; cq1[j] = ...; ... cq8[j] = ...;
    }
  }
}
```

**关键属性：**

- `__restrict__` 告知编译器 18 个指针互不别名 → 解锁 j 循环自动向量化
- `cp_k[j]` 是 j 方向 stride-1 load → SIMD 宽向量加载（无 gather）
- 无 `#pragma omp simd`：避免强制 `vdivpd zmm`（低吞吐量），让编译器自主决策

### 2.3 collision 分发

```cpp
if (impl == COLLISION_MULTICELL) {
    collision_multicell(mesh_out, mesh_in);
} else if (impl == COLLISION_UNROLLED) {
    // ... per-cell 内层 j 循环
} else {
    // ... baseline
}
```

## 4. 分析

### 4.1 collision 加速来源

multicell vs unrolled，collision 一致快 10–13%。主要机制：

1. **消除 k*W*H 乘法**：per-cell 路径每次 `Mesh_get_f` 都要计算
   `k*W*H + x*H + y`；multicell 只在 i 循环开头计算一次 `col_off = i*H`，
   然后 `cp_k[j]` 是简单的指针偏移

2. **更好的 ILP**：hoisted plane pointer 使编译器可以更早发射 load 指令，
   减少 load-use latency 对计算吞吐量的影响

**注意：** 本节分析只对应优化21正式实现 `multicell`。后续优化22对其做了显式向量化验证，并进一步引入 `multicell_vec`，详见 `优化22.md`。

### 4.2 np=4 收益更大（+16.1%）

np=4 下 collision 和 halo 占比相近（约 38% vs 34%）。multicell 将
collision 从 2.24s 降至 2.01s，每步省出 0.23s，MPI 进程同步后
halo 等待时间也有所缩短（1.57s vs 2.00s），形成正反馈。
halo_finish 时间缩短 0.43s，部分原因是 MPI 调度的 run-to-run 方差，
部分是实际等待减少（进程间更均匀地完成 collision 步骤）。

### 4.3 np=4 halo_finish 仍是次要瓶颈

```text
np=4, k-outer + multicell:
  collision = 39.95%（2.01s）
  halo_finish = 31.22%（1.57s）
    ├── halo: vert_pack   = 8.24%（0.41s）
    ├── halo: waitall     = 16.83%（0.85s）
    └── halo: vert_unpack = 14.30%（0.72s）
```

`vert_pack + vert_unpack = 1.13s = 22.5%`，是 pack/unpack 的主要成本。
注：后续优化22中的 `multicell_vec` 会进一步改变此瓶颈比例；详见 `优化22.md`。

### 4.4 优化21的正式结论边界

本轮可以下的正式结论是：

- 在 **SoA k-outer propagation 已固定** 的前提下，`multicell collision` 明显优于 `unrolled collision`
- 收益来自：
  - plane pointer hoist
  - 列内 `j` 方向 stride-1 连续访问
  - 编译器对多 cell 循环的自动向量化

本轮**不直接声称**：

- `multicell` 已经是整个工程的最终最优 collision 形式
- 或者它已经替代了所有历史基线的正式比较口径

它的正确定位是：

- 优化20之后的下一阶段 collision kernel 重组结果
- 优化22继续处理通信瓶颈前的最新正式实现

## 6. 瓶颈转移与优化22方向

基于优化21正式实现 `multicell`，下一步优先级如下：

| 配置 | 当前主要瓶颈 | 次要瓶颈 |
| --- | --- | --- |
| `np=1, OMP=4` | collision（76.73%） | propagation_interior（约 14%） |
| `np=2, OMP=4` | collision（57.58%） | halo_finish（16.07%） |
| `np=4, OMP=4` | collision（39.95%）+ halo_finish（31.22%） | vert_pack/unpack（22.5%） |

**优化22建议（按优先级重新排序）：**

1. **collision_multicell 的显式向量化 / AVX 验证**

   当前 `multicell` 仍主要依赖：
   - plane pointer hoist
   - `j` 方向 stride-1 连续访问
   - 编译器自动优化

   下一步应先验证：
   - `j` 内层是否真的已经自动向量化
   - 若未向量化，根因是什么
   - 显式 AVX / AVX-512 是否值得保留为新主路径

2. **vertical halo pack/unpack 格式改造**

   当前 pack 格式是 cell-interleaved，每 cell 从 9 个 plane gather 1 个值 →
   9 次散乱 load。候选方案：**k-major pack**，每个方向连续扫描一行：

   ```cpp
   for (int k = 0; k < 9; k++) {
       memcpy(buf + k * inner_w,
              Mesh_get_plane(mesh, k) + 1*H + j_lo, inner_w * sizeof(double));
   }
   ```

   此格式 pack/unpack 均变为 9 次 stride-1 memcpy，双端同步修改。

3. **propagation_interior 在 np=1 下的优化**

   当前 `propagation_interior` 已是 nowait + memcpy 结构。若下一轮先完成
   collision 向量化，np=1 下 propagation 占比将进一步上升，届时再判断是否还有优化空间。
