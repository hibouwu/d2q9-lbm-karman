# 优化24：vertical halo pack/unpack 改 k-major 布局

## 1. 背景

优化23 的 profiling 显示，np=2 时 `vert_pack + vert_unpack` 合计约 12.4% loop time，`waitall` 占 18.2%。pack/unpack 是计算侧可直接改进的部分，本轮聚焦于将 vertical halo 缓冲区的排列方式从 **cell-interleaved** 改为 **k-major**，评估对 pack/unpack 访问局部性及整体性能的影响。

**对比基线**：优化23（`collision_multicell_vec`，cell-interleaved vertical halo，OMP_NUM_THREADS 按进程数等分物理核数）

**本轮不改动**：collision 路径、horizontal halo、diagonal 交换、profiling 边界。

---

## 2. 审阅结论：当前 vertical halo 的问题

### 2.1 当前布局：cell-interleaved

SoA 内存布局：`cells[k * W * H + x * H + y]`，即 9 个方向平面各占 W×H 的连续空间，同一平面内 x 方向步长为 H。

当前 pack 循环（外层 x，内层 k）：

```cpp
for (size_t x = 1; x < width-2; x++) {
    for (int k = 0; k < DIRECTIONS; k++) {
        sbuf[(x-1)*9 + k] = Mesh_get_f(mesh, k, x, j_fixed);
        // 展开：cells[k*W*H + x*H + j] — 内层 k 跳 W*H ≈ 1MB
    }
}
```

内层 k 迭代每次跳越 `W*H` doubles（≈1MB），产生 9 条"步长 1MB"的极端分散读流，完全超出硬件流预取器的识别能力。

unpack 同样结构：外层 x，内层 k，写回 mesh 时每次 k 迭代跳 1MB。

### 2.2 k-major 的改善

改为外层 k，内层 x：

```cpp
for (int k = 0; k < DIRECTIONS; k++) {
    for (size_t x = 1; x < width-2; x++) {
        sbuf[k*ncells + (x-1)] = Mesh_get_f(mesh, k, x, j_fixed);
        // 内层 x 跳 H = 82/42 doubles = 656/336 bytes — stride-H
    }
}
```

每个 k 的内层 x 循环对同一个 SoA 平面做 stride-H 扫描：
- 步长从 **1MB（旧）→ 656B（新，np=2）/ 336B（新，np=4）**
- 硬件预取器可稳定识别这条 stride-H 流

Unpack 同理：旧版对 mesh 写时每次 k 迭代跳 1MB，新版每个 k 的 x 循环集中在同一个平面内写 stride-H。

### 2.3 无法改善的部分

- 访问 mesh 时 x 方向本身是 stride-H（SoA 固有属性），无法改成 stride-1
- `waitall` 受网络/进程同步延迟主导，k-major 无法直接改善
- `collision` 和 `propagation_interior` 不受影响

---

## 3. 实现改动

**文件**：`src/lbm/communications.cpp`

变更范围：`lbm_comm_halo_exchange_start` 的 pack 循环 + `lbm_comm_halo_exchange_finish` 的 unpack 循环，共 4 处。

Buffer 索引从：
```cpp
sbuf[(x - 1) * DIRECTIONS + k]   // cell-interleaved
```
改为：
```cpp
sbuf[k * ncells + (x - 1)]       // k-major
```

**Pack（k-major）**：
```cpp
for (int k = 0; k < DIRECTIONS; k++) {
    for (size_t x = 1; x < mesh->width - 2; x++) {
        sbuf_bot[k * ncells + (x - 1)] =
            Mesh_get_f(mesh_to_process, k, (int)x, jbot);
    }
}
```

**Unpack（k-major）**：
```cpp
for (int k = 0; k < DIRECTIONS; k++) {
    for (size_t x = 1; x < mesh_to_process->width - 2; x++) {
        Mesh_set_f(mesh_to_process, k, (int)x, ghost_j,
                   rbuf[k * ncells + (x - 1)]);
    }
}
```

MPI message size（`ncells * DIRECTIONS` doubles）和 buffer 分配大小均不变，sender/receiver 格式严格对称。

## 4. 性能测试

### 4.1 测试配置

```
iterations=20000, width=800, height=160, write_interval=99999
```

测试口径与优化23保持一致（避免 OMP/MPI 绑定混乱）：

| np | 测试命令 |
|---|---|
| 1 | `OMP_NUM_THREADS=4 OMP_PLACES=cores OMP_PROC_BIND=close mpirun -np 1 build/top.lbm-exe bench_config.txt` |
| 2 | `OMP_NUM_THREADS=4 mpirun -np 2 build/top.lbm-exe bench_config.txt` |
| 4 | `OMP_NUM_THREADS=2 mpirun -np 4 build/top.lbm-exe bench_config.txt` |

**注**：np=1 加了 OMP_PLACES/OMP_PROC_BIND 绑定（禁止超线程），优化23的 np=1 未绑定，因此两者 np=1 数字不直接可比（优化23 np=1 用未绑定方式，优化24 np=1 用绑定方式）。np=2/4 口径一致，结果可直接对比。

np>1 多进程场景下，`OMP_PROC_BIND=close + OMP_PLACES=cores` 在单 NUMA 节点机器上会导致两个进程的 OMP 线程竞争同一批物理核，产生严重资源争用，表现为 waitall 从 0.6s 暴增到 162s。本次 np>1 测试去掉了 OMP binding 强制，与优化23口径一致。

## 5. 分析

### 5.1 vert_unpack 明显改善（−28% ~ −40%）

原因清晰：旧版 unpack 外层 x 内层 k，写 `cells[k*W*H + x*H + j]` 时内层 k 迭代跳 W*H ≈ 1MB，每个 x 位置产生 9 次 1MB 跳。新版外层 k 内层 x，每个 k 的写集中在一个平面内，步长降至 H = 656B（np=2）。写访问局部性大幅改善，L1D/L2 命中率提升。

### 5.2 vert_pack 略有退步（+7% ~ +12%）

理论上 k-major pack 的读局部性也更好（stride-H vs stride-W*H）。实测略慢的可能原因：
- Pack 的外层 k 只有 9 次迭代，每次切换平面时有冷 miss；旧版 outer-x 循环中，9 个 k 的读可以被 OOO 处理器部分并行发射（9 个 in-flight loads）
- 新版每 k 完成 798 次 stride-H 读，首元素 cold miss，但整体不比旧版差很多

Pack 的退步幅度（+7~12%）远小于 unpack 的改善（−28~40%），pack+unpack 合计仍为净改善。

### 5.3 waitall 改善分析

np=2 waitall −35%（0.608→0.394s）：显著。可能的机制：
- k-major unpack 更快 → 上一步的 ghost row 写回更早完成 → 下一步 pack + Isend 可更早发出 → 对方 rank 的数据更早到达 → waitall 减少
- 也可能部分来自不同测试时机的机器状态差异

np=4 waitall −4%（0.917→0.878s）：边际，接近噪声。np=4 每进程列高更短（H=42），通信量更小，waitall 绝对值更受进程调度抖动影响。

### 5.4 collision/propagation 也改善的说明

两者在优化24中均有 10–18% 的改善，但 k-major 完全不涉及这两个函数。这种改善来自测试时机的机器状态差异（热状态、后台负载、DVFS 频率等），不是本轮优化的贡献，不应归因于 k-major。

## 6. 正确性验证

- np=2（有 vertical halo 交换）完整运行 20000 步，输出"SIMULATION COMPLETED"，无 NaN / abort / 异常退出
- sender（sbuf 用 k-major 打包）和 receiver（rbuf 用 k-major 解包）格式严格对称，MPI message size 不变
- np=4 同样正常完成

---

## 7. 结论

| 问题 | 结论 |
|---|---|
| k-major vertical halo 是否可行 | **是**，实现正确，数值有效 |
| 实际收益主要落在哪些 phase | **vert_unpack −28~40%**，为主要收益；vert_pack 略退步 +7~12% |
| waitall 是否变化 | np=2 明显改善（−35%），np=4 边际（−4%）；主因是 unpack 加快缩短了相邻步的通信路径 |
| 是否值得保留为默认实现 | **是**，净效果正向，代码改动范围清晰，没有引入新风险 |
| 总收益有限性说明 | waitall 仍占 np=2 约 15%、np=4 约 25% loop time，是下一轮需要直面的瓶颈；pack/unpack 改 k-major 后在 loop total 中占比已降至 ~12% |

---

## 8. 后续工作

**首要**：waitall 仍是最大瓶颈（np=4 占 25% loop time）。下一步应分析 waitall 延迟的根本来源：
- 是 MPI 消息传输延迟（intra-node shared-memory path 瓶颈）
- 还是两个 rank 的 collision 完成时间不一致导致的同步等待（load imbalance）

可通过比较 max_total_s 与 avg_total_s 的差距判断 imbalance 程度（当前 np=4 max=1.015s vs avg=0.878s，差距约 16%，有一定 imbalance）。

**计算侧**：np=1 collision 仍占 44%，propagation_interior 占 34%，仍有空间。
