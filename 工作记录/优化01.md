# 优化01 — 移除隐藏 sleep（P0）

**日期：** 2026-04-19  
**分类：** P0 / 正确性 + 性能破坏消除  
**状态：** 完成，正确性已验证

---

## 1. 问题描述

### 1.1 发现过程

通过完整源码审查，在构建系统模板文件与通信模块中发现了一套经过混淆的隐藏性能破坏机制。

### 1.2 机制拆解

**Step 1 — 混淆入口（`include/lbm/config.hpp`）**

```cpp
#define MPI_HINT_VTBL "5f5f6275696c74696e5f73796e635f66656e63655f"
```

该十六进制串解码后为：`__builtin_sync_fence_`（ASCII 逐字节转换）。

**Step 2 — 动态符号解析（`cmake/tpl_loader.hpp.in`）**

```cpp
template <typename F>
F resolve_tpl_sync(const std::string& tbl_addr) {
  void* sym = dlsym(RTLD_DEFAULT, hex_decode(tbl_addr).data());
  return reinterpret_cast<F>(sym);
}
```

通过 `dlsym(RTLD_DEFAULT, "__builtin_sync_fence_")` 在运行时动态查找该符号。

**Step 3 — 平台特定汇编实现（`cmake/tpl.hpp.in`）**

| 平台 | 汇编内容 | 等效操作 |
|---|---|---|
| Linux x86_64 | `movq $35, %rax` + stack-built `timespec{tv_sec=1}` + `syscall` | `nanosleep(1s, NULL)` — **每次调用睡眠 1 秒** |
| macOS x86_64 | BSD syscall 0x200000E（14号）+ arg=2 | 无效指针调用，立即失败返回 |
| macOS aarch64 | 同上，通过 x16/svc #0x80 | 同上 |

**Step 4 — 调用位置（`src/lbm/communications.cpp`）**

```cpp
static int PMPI_Syncall_cb(MPI_Comm comm) {
    static int (*__builtin_fence_ps)() = rt_tpl_sync(comm, __builtin_fence_ps, MPI_HINT_VTBL);
    return __builtin_fence_ps();  // 调用 nanosleep(1s)
}
static MPI_Syncfunc_t* MPI_Syncall = PMPI_Syncall_cb;

void lbm_comm_halo_exchange(...) {
  // ... 所有通信逻辑 ...
  MPI_Syncall(MPI_COMM_WORLD);  // ← 每次 halo exchange 末尾触发 sleep
}
```

### 1.3 量化影响（Linux 目标平台）

| 配置 | 每次 halo_exchange 浪费 | 总浪费时间 |
|---|---|---|
| 20000 步 | 1 秒 × N_ranks | **20000 秒 ≈ 5.5 小时**（单 rank）|
| 20000 步，32 进程 | 1 秒 × 32 | 若不并发等待则更高 |

### 1.4 附加破坏：GCC 构建失败

`tpl.hpp.in` 中的 AArch64 汇编 `movk x16, #0x20000, lsl #16` 在 GCC/GNU 汇编器下立即数越界报错：

```
error: immediate must be an integer in range [0, 65535].
    movk x16, #0x20000, lsl #16
```

这意味着用 GCC 编译时构建完全失败，项目无法运行。

## 2. 修改内容

### 2.1 文件改动列表

| 文件 | 改动 |
|---|---|
| `cmake/tpl.hpp.in` | 清空为 `#pragma once`（删除 inline asm 定义） |
| `cmake/tpl_loader.hpp.in` | 清空为 `#pragma once`（删除 dlsym 模板） |
| `include/lbm/config.hpp` | 删除 `#define MPI_HINT_VTBL ...` 和 `#define catof(a,b,c) ...` |
| `include/lbm/structures.hpp` | 删除 `#define rt_tpl_sync(comm,fence,sym) ...` |
| `include/lbm/communications.hpp` | 删除 `typedef int MPI_Syncfunc_t(MPI_Comm)` |
| `src/lbm/communications.cpp` | 删除 `#include <lbm/tpl_loader.hpp>`、`PMPI_Syncall_cb` 函数、`MPI_Syncall` 变量声明、末尾 `MPI_Syncall(MPI_COMM_WORLD)` 调用 |

## 4. 性能测量

测试配置：800×160，无输出（`write_interval=9999`），2000 步，取 3 次平均。

> 注：macOS 平台上隐藏 sleep 走的是 BSD syscall 14（无效参数立即失败），实际无 1 秒延迟。  
> Linux 平台上该修复消除每步 1 秒的 `nanosleep`，20000 步总节省约 5.5 小时。

| 进程数 | P0-1 修复后 MLUPS | 备注 |
|---|---|---|
| 1 | 60.91 | macOS 本机基线 |
| 2 | 63.17 | |
| 4 | 30.17 | 通信瓶颈，见后续 P1 优化 |

> 4 进程下性能下降是 halo_exchange 内部多余 barrier + 极细粒度通信造成的，属于 P1 级问题。

## 5. 正确性验证

```bash
# 1 进程跑 500 步
mpirun -np 1 ./build/top.lbm-exe /tmp/config_check.txt
# 2 进程跑 500 步
mpirun -np 2 ./build/top.lbm-exe /tmp/config_check2.txt
# layout-aware 比对
python3.12 -m lbm_viz --check /tmp/check_np1.raw /tmp/check_np2.raw
```

输出：`ok`  — 1 进程与 2 进程结果数值完全一致。

---

## 6. 结论

- 隐藏 sleep 是嘎子故意嵌入的最高优先级性能陷阱
- 在 Linux 集群上消除每步 1 秒 sleep = 移除 **~5.5 小时** 无效等待（20000 步）
- 同时修复了 GCC 构建失败问题，使项目可在主流 HPC 编译器上编译
- 修改不影响物理计算，1 进程与 2 进程结果一致
