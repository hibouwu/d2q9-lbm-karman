## 1. AoS 布局的问题

当前 AoS 布局为 `cells[(x * H + y) * 9 + k]`：

- 每个 cell 的 9 个方向连续存储
- **对 collision**：读写 cell(x,y) 的 9 个方向是 72 字节连续访问，1-2 个 cache line，对编译器 SIMD 友好
- **对 propagation**：`propagate_cell(i,j)` 需要从 9 个不同位置（i-dk_x, j-dk_y）读，这 9 个来源彼此相邻但不连续
- **对 AoS 的本质限制**：内层 j 循环访问步长为 `9 * sizeof(double) = 72 bytes`，而不是步长 1

---

## 2. SoA 设计

### 2.1 内存布局

`cells[k * W * H + x * H + y]`

- 第一维：direction plane（k=0..8），9 个独立的 W×H 连续平面
- 第二维：x（列，slower）
- 第三维：y（行，faster），与当前循环结构兼容

**SoA 的理论优势：**

- 对 propagation（k-outer 循环）：固定 k 的一次扫描只访问一个平面，stride-1 读写，SIMD-ready
- 对 collision（scalar gather from 9 planes）：9 个方向各自在独立平面上，内层 j 循环内每个方向是 stride-1 访问，便于多流预取

### 2.2 新访问器（`include/lbm/structures.hpp`）

```cpp
// SoA: cells[k * W * H + x * H + y]
static inline double Mesh_get_f(const Mesh* mesh, int k, int x, int y);
static inline void   Mesh_set_f(Mesh* mesh, int k, int x, int y, double v);
static inline double* Mesh_get_plane(const Mesh* mesh, int k);
```

废弃的旧接口：`Mesh_get_cell`、`Mesh_get_col`（已从头文件中移除）。

### 3.2 通信层适配

**Horizontal（列交换）：**

旧版直接传 `Mesh_get_col(mesh, x)`（连续内存块）。SoA 改为显式 pack/unpack：

```cpp
// pack: for j=1..height-2, for k=0..8: buf[(j-1)*9+k] = Mesh_get_f(mesh, k, x, j)
// send/recv
// unpack: Mesh_set_f(mesh, k, x, j, buf[(j-1)*9+k])
```

**Diagonal（角点单 cell）：**

旧版传 `Mesh_get_cell(mesh, x, y)`（9 个连续 double）。SoA 改为 pack/unpack 到栈上 `double cell_buf[9]`。

**Vertical pack/unpack（已有显式 memcpy，现改为逐值）：**

```cpp
// 旧: memcpy(sbuf + (x-1)*9, Mesh_get_cell(mesh, x, row), 9*sizeof(double))
// 新: for k=0..8: sbuf[(x-1)*9+k] = Mesh_get_f(mesh, k, x, row)
```

Pack/unpack buffer 格式不变（cell-interleaved），MPI 消息内容语义一致。

### 3.3 `compute_cell_collision_unrolled` 同步更新

AoS 实验版已同步改为 SoA 版本（原有标量 load/store 结构天然适配）：

### 5.2 关键技术结论：为什么 SoA baseline 不加 `#pragma omp simd`

初步审查曾怀疑 SoA baseline `compute_cell_collision()` 丢失了优化16中的 `#pragma omp simd`，会导致与 AoS baseline 口径不一致。实测后确认这一判断是错误的。

原因如下：

- **AoS**：`cell_in[k]`（`k=0..8`）是 9 个连续 double，`#pragma omp simd reduction` 可触发有效的连续向量加载
- **SoA（当前 per-cell 结构）**：`Mesh_get_f(mesh, k, x, y)` 对 `k` 的访问步长是 `W*H`，即跨 plane gather 访问
- 在这种访问模式下，显式 `#pragma omp simd` 会诱导编译器生成 gather 指令，反而显著变慢

实测结论：

- 对 SoA per-cell `collision`，**不加 pragma 是正确选择**
- 这不是遗漏，而是针对 SoA 访问模式做出的有意识实现决策

## 7. 性能分析：为何 SoA baseline 比 AoS 慢

### 5.1 核心问题：当前 kernel 结构未充分利用 SoA

SoA 的优势需要 **k-outer / (i,j)-inner** 的循环结构才能完全发挥。当前实现保留了 **per-cell** 结构（外层 i, 内层 j，最内层 k 在 `propagate_cell/compute_cell_collision` 内），这正是 SoA 的"反模式"：

**propagation（最大回退来源）：**

- AoS：`propagate_cell(i,j)` 写 `cell(i,j)` 的 0..8 方向 = 72 字节连续写，读 9 个相邻源 cell（各 72 字节，空间紧凑）
- SoA：`propagate_cell(i,j)` 分别写 9 个相距 `W*H*8 ≈ 1MB` 的 plane 位置，读 9 个相距 1MB 的 plane 位置

对于 W=802, H=162：每个 plane = 802×162×8 ≈ 1MB。访问 9 个 plane = 9MB，不适合 L2/L3 cache。AoS 的同一列数据约 11KB，9 个邻居列约 100KB，完全在 L2 内。

**halo pack/unpack 回退：**

- AoS pack：`memcpy(9 doubles)` 一次性拷贝 72 字节（CPU 可用 SIMD store）
- SoA pack：9 次独立 `Mesh_get_f(...)` load，各自来自不同 plane，无法合并

### 5.2 为何 collision 在某些配置下更快

对于 np=1 和 np=2，SoA collision 比 AoS 稍快（-6%/-3%）：

- AoS collision 内层 j 循环：读 `cell_in[k*72字节步长]`（stride-9）
- SoA collision 内层 j 循环：读 `plane_k[i*H + j]`（stride-1）

SoA 每个 plane 内是 stride-1 读写，硬件预取器可同时跟踪 9 个 stride-1 流，优于 AoS 的 stride-9 访问。但这仅在 np=1/2（每 rank 数据量大、cache pressure 高）时显现；np=4 时每 rank 数据小，AoS 整个列完全在 L1。

### 5.3 结论

**当前 SoA baseline（per-cell 结构）比 AoS 慢，是预期内的结果。** SoA 需要配合正确的循环结构重组才能盈利。这正是优化20（SoA unrolled + k-outer 循环重组）的设计目标。

## 10. 优化20 的任务定义

根据本次测试结论，**优化20（SoA unrolled）需要完成以下两件事才能超越 AoS**：

### 8.1 必须做：propagation 循环重组（k-outer）

将 `propagation_interior / propagation_border / propagation` 改为 k-outer 结构：

```cpp
// k-outer SoA propagation:
for (int k = 0; k < DIRECTIONS; k++) {
    const int dk_x = (int)direction_matrix[k][0];
    const int dk_y = (int)direction_matrix[k][1];
    double* plane_out = Mesh_get_plane(out, k);
    const double* plane_in  = Mesh_get_plane(in,  k);
    // For each (i,j): plane_out[i*H+j] = plane_in[(i-dk_x)*H + (j-dk_y)]
    // 内层 i/j 循环访问 plane 内 stride-1 → SIMD-ready
}
```

这样每个 k 只访问 **1 个 plane**，完全消除跨 plane 的 cache 压力。

### 8.2 可选：halo pack 优化

垂直 halo pack 改为 k-outer：先把一整行 `plane_k[x=1..W-2, j=row]` 连续打包：

```cpp
for (int k = 0; k < DIRECTIONS; k++) {
    const double* plane = Mesh_get_plane(mesh, k);
    for (int x = 1; x <= ncells; x++) {
        sbuf[k * ncells + (x-1)] = plane[x * H + row];  // or cell-interleaved
    }
}
```

注意：若改变 pack 格式，双端的 pack/unpack 必须同步修改。

## 12. 最终结论

优化19完成了两件事：

1. **完成 AoS → SoA 的存储布局重构**
2. **形成了经审查修正后的正式 SoA baseline（v1.1）**

这版文档保留了 **v1.0 → v1.1** 的完整演进过程：

- `v1.0`：首次完整 SoA benchmark，暴露性能形态
- `v1.1`：修正会影响正式对比可信度的实现细节，作为正式基线定稿

最终结论如下：

- **SoA baseline v1.1 可作为可信正式比较基线**
- 相对 AoS baseline，SoA baseline 在 `np=1/2/4, OMP=4` 下分别回退 **8.8% / 22.4% / 29.2%**
- 主要原因不是 SoA 实现错误，而是：
  - `propagation` 仍保留 per-cell / k-inner 结构
  - halo pack/unpack 仍是逐 direction gather
- `collision` 路径中不加 `#pragma omp simd` 是经过验证的正确选择

因此，优化20的目标已经非常明确：

1. 将 `propagation` 重组为 **k-outer SoA kernel**
2. 在 SoA 布局稳定基础上，再比较 **SoA baseline** 与 **SoA unrolled**
