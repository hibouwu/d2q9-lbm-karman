# 优化26：延长 overlap 窗口——单一 OMP parallel 区域重构

## 1. 背景

优化25 §10 明确了正确的延长 overlap 方案，并说明了为什么需要单一 `#pragma omp parallel` 区域：

- 当前 overlap 窗口仅覆盖 `propagation_interior`（≈ 0.019ms/call），而 waitall ≈ 0.016ms/call（np=2），两者差距已很小
- 优化25 §10 的拆分实验（`collision_border` 串行 + `collision_interior` 并行）导致 FOM −34%，根因是每步两次 OMP worker 休眠/唤醒，代价约 100μs
- **正确方向**：将 border collision → halo_start → interior collision → propagation_interior → halo_finish → propagation_border 放进一个 `#pragma omp parallel` 区域，用 `omp for`/`omp master`/`omp barrier` 协调，避免跨函数的 parallel 重入开销

本轮目标：实现该重构，使 overlap 窗口扩展为 interior collision + propagation_interior，理论上可完全掩盖 waitall。

---

## 2. 实现方案

### 2.1 新增接口

为避免在热路径上重复 parallel 区域，引入三个内部接口，**调用者必须已处于 `#pragma omp parallel` 区域内**：

```cpp
// physics.hpp
// i=1..W-2, j∈[j_begin, j_end)，用 omp for（有隐式 barrier）
void collision_rows(Mesh* out, const Mesh* in, int j_begin, int j_end);

// 等价于 propagation_interior，但无 omp parallel 包装，末尾加显式 barrier
void propagation_interior_omp_region(Mesh* out, const Mesh* in, bool has_vn);

// 等价于 propagation_border，但无 omp parallel 包装，末尾加显式 barrier
void propagation_border_omp_region(Mesh* out, const Mesh* in);
```

### 2.2 collision_rows 实现

默认路径（`multicell_vec`）提取为 `collision_multicell_vec_rows_omp`，与原 `collision_multicell_vec` 内核完全相同，仅将 `#pragma omp parallel for` 替换为 `#pragma omp for`：

```cpp
static void collision_multicell_vec_rows_omp(
    Mesh* out, const Mesh* in, int j_begin, int j_end)
{
  // ... 同 collision_multicell_vec 的 plane pointer 设置 ...
#pragma omp for schedule(static)
  for (int i = 1; i < W - 1; i++) {
    // ... 同原内核，j 循环范围改为 [j_begin, j_end) ...
  }
  // omp for 隐式 barrier
}
```

`collision_rows` 做 impl dispatch：multicell/vec/vec_pf 均走 `collision_multicell_vec_rows_omp`；baseline/unrolled 走逐单元 fallback（同样用 `omp for`）。

`j_begin >= j_end` 时提前 return，所有线程一致 return，不会产生孤立的 `omp for`。

### 2.3 propagation_*_omp_region 实现

将原函数体中的 `#pragma omp parallel { ... }` 去掉外层包装，直接暴露内部循环。`nowait` 保留（k 方向流水），在函数末尾补一个显式 `#pragma omp barrier`：

```cpp
void propagation_interior_omp_region(...) {
  for (int k = 0; k < DIRECTIONS; k++) {
    ...
#pragma omp for schedule(static) nowait
    for (int i = ...) { ... }
  }
#pragma omp barrier  // 同步，确保所有写入完成再返回
}
```

### 2.4 main.cpp：has_vn 路径重构

`!has_vn` 路径保持原结构不变，仅 `has_vn` 路径进入单一 parallel 区域：

```cpp
} else {
  const int H = (int)temp.height;
#pragma omp parallel
  {
    // 1. border collision（j=1, j=H-2）
#pragma omp master
    { LBM_PROF_BEGIN(PROF_COLLISION); }
    collision_rows(&temp, &mesh, 1, 2);       // omp for，隐式 barrier
    collision_rows(&temp, &mesh, H-2, H-1);   // omp for，隐式 barrier

    // 2. halo_start（仅 master，non-master 立即进入步骤 3）
#pragma omp master
    {
      LBM_PROF_BEGIN(PROF_HALO_START);
      lbm_comm_halo_exchange_start(&mesh_comm, &temp);
      LBM_PROF_END(PROF_HALO_START);
    }
    // 无 barrier：interior collision 不依赖 ghost row

    // 3. interior collision（j=2..H-3，与 MPI 传输重叠）
    collision_rows(&temp, &mesh, 2, H-2);     // omp for，隐式 barrier
#pragma omp master
    { LBM_PROF_END(PROF_COLLISION); }

    // 4. propagation_interior（与 MPI 传输重叠）
#pragma omp master
    { LBM_PROF_BEGIN(PROF_PROPAGATION_INTERIOR); }
    propagation_interior_omp_region(&mesh, &temp, true);  // 末尾有显式 barrier
#pragma omp master
    { LBM_PROF_END(PROF_PROPAGATION_INTERIOR); }

    // 5. halo_finish（仅 master）
#pragma omp master
    {
      LBM_PROF_BEGIN(PROF_HALO_FINISH);
      lbm_comm_halo_exchange_finish(&mesh_comm, &temp);
      LBM_PROF_END(PROF_HALO_FINISH);
    }
#pragma omp barrier  // 确保 ghost row 写入完成再做 border propagation

    // 6. propagation_border（j=1, j=H-2）
#pragma omp master
    { LBM_PROF_BEGIN(PROF_PROPAGATION_BORDER); }
    propagation_border_omp_region(&mesh, &temp);  // 末尾有显式 barrier
#pragma omp master
    { LBM_PROF_END(PROF_PROPAGATION_BORDER); }
  } // end #pragma omp parallel
}
```

### 2.5 profiling 线程安全

原 `LBM_PROF_BEGIN/END` 写全局 `s_t0[]`/`s_total[]`，多线程调用存在 race。`has_vn` 路径内所有 `LBM_PROF_*` 调用全部包裹在 `#pragma omp master { }` 中，`!has_vn` 路径（单线程段）维持原调用方式。无需改动 profiling 实现本身。

## 6. 结果分析

### 6.1 FOM 基本持平（+0.3%）

`loop total` 从 0.1591ms → 0.1586ms，FOM 从 800.27 → 802.31 MLUPS，提升在测量噪声范围内。**overlap 在 intra-node 场景下收益接近零**，原因见 §6.3。

### 6.2 collision % 暴增是计时语义变化，非退化

新版 `PROF_COLLISION` 的计时窗口从 border collision 开始、到 interior collision 结束，**中间包含了 halo_start 的执行时间**（约 14%），因此显示 44% ≈ 旧版 26% + halo_start 14%。实际计算量完全相同。

### 6.3 intra-node MPI 下 overlap 收益为何接近零

优化25 §6.3 已归因：intra-node shared-memory MPI 下，`Isend` 本身即完成拷贝（eager protocol），传输延迟 < 0.003ms。真正的 waitall 等待时间是两个 rank 进入 Waitall 的时间差（调度抖动），而非网络传输时间。即使 overlap 窗口扩大，也无法消除这种调度抖动导致的等待。

**overlap 优化的实际收益场景**：跨节点通信（延迟 ≥ 1ms），此时 interior collision + propagation_interior（≈ 0.07ms/call）可完全覆盖消息往返时延，waitall 降至 0。本次测试环境无法体现该收益。

### 6.4 halo: vert_unpack 下降（−4.5%）

从 0.683s → 0.540s（avg），表现好于预期。可能原因：在 parallel 区域内，`halo_finish`（master）发起 unpack 时，其他 worker 线程已完成 propagation_interior 并处于 barrier 等待状态，OS 调度给 master 的资源更充足，unpack（纯串行 memcpy 循环）完成更快。

### 6.5 halo: waitall 略有上升（+2.7%）

从 0.246s → 0.331s。与 §6.3 的分析一致：intra-node 下 waitall 主要是调度抖动，并行区域结构改变了 worker 线程在步骤间的调度时序，使两 rank 进入 Waitall 的时间差略有增大。属正常波动，不影响结论。

## 7. 结构正确性说明

### 7.1 barrier 布局

| 位置 | 类型 | 必要性 |
| --- | --- | --- |
| 每个 `collision_rows` 末尾 | omp for 隐式 barrier | border 行写入完成后 master 才能打包发送 |
| 步骤 2→3 之间 | 无 barrier | interior collision 不依赖 ghost row，intentional |
| `propagation_interior_omp_region` 末尾 | 显式 barrier | 确保所有 propagation 写入完成后 master 才调 halo_finish |
| halo_finish 之后 | 显式 `#pragma omp barrier` | ghost row 写入后 propagation_border 才能读 |
| `propagation_border_omp_region` 末尾 | 显式 barrier | 确保 border 写入完成后退出 parallel 区域 |

### 7.2 MPI 线程安全

所有 MPI 调用（`halo_exchange_start`、`halo_exchange_finish`）均在 `#pragma omp master` 块内执行，保证单线程调用。MPI_Init 以 `MPI_THREAD_FUNNELED` 兼容的方式使用。

## 8. 结论

| 问题 | 结论 |
| --- | --- |
| OMP 线程唤醒开销 | 已消除：整个 collision→propagation 在单一 parallel 区域内执行 |
| overlap 窗口 | 已从 propagation_interior（0.019ms）扩展至 interior_collision + propagation_interior（≈ 0.071ms） |
| intra-node FOM 变化 | +0.3%，在噪声范围内；intra-node MPI 延迟不是瓶颈，overlap 收益无法体现 |
| 数值正确性 | 逐字节验证通过（np=1，20000步） |
| 代码结构 | !has_vn 路径不变；has_vn 路径重构为单一 parallel 区域，为多节点基准测试就绪 |

## 9. 后续工作

### 最高优先级：多节点基准测试

当前结论受限于 intra-node 测试环境。在真实集群（MPI 延迟 ≥ 1ms）上测试，是验证本轮重构实际收益的唯一方法。预期：halo_finish（特别是 waitall）时间显著下降，FOM 有实质提升。

### 次优先级：结构性不均衡（edge vs middle rank）

np≥3 时 vert_pack max/min ≈ 2×，edge rank 在 Waitall 等待 middle rank 的时间不可避免。可通过非均匀行分割（给 middle rank 分配稍少行数）来均衡负载，但需改动 mesh 分割逻辑。

### 可选：collision_rows 的 vec_pf 路径

当前 `collision_rows` 对 `COLLISION_MULTICELL_VEC_PF` 走 vec 路径（无 prefetch）。如需保留 prefetch 效果，需单独实现带 `__builtin_prefetch` 的 `_rows_omp` 变体。对 border 行（仅 2 行）效果有限；对 interior 行可能有边际收益。
