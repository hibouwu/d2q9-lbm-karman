# 优化18 — Phase-level Profiling（可观测性增强）

**日期：** 2026-04-26  
**分类：** 工具 / 可观测性  
**状态：** 完成，编译验证通过（开启/关闭两路均可构建）

---

## 1. 背景

### 1.1 当前可观测性的局限

P0～P3 的优化完成后，程序在主循环结束时只输出一个粗粒度指标：

```
FOM:  65.7 MLUPS
```

这个数字能回答"整体吞吐量是多少"，但无法回答：

- 计算瓶颈在哪个 phase（`collision` 还是 `propagation_interior`）？
- 通信开销（`halo_start` + `halo_finish`）占总循环时间的比例是多少？
- MPI `Waitall` 等待了多久（体现 rank 间负载均衡）？
- 在集群上，哪个 rank 最慢（体现 I/O 或网络不均衡）？

在不同的进程数和 mesh 大小下，这些问题的答案完全不同。没有 phase-level 数据，就无法判断下一步优化的方向，也无法定量验证现有优化的收益分布。

### 1.2 为什么当前只有 MLUPS 不够

| 问题 | MLUPS 能回答？ |
|---|---|
| 哪个 phase 是热点？ | ❌ |
| 通信时间有多少被计算隐藏？ | ❌ |
| 最慢的 rank 比最快的慢多少？ | ❌ |
| I/O 写入开销是否影响 benchmark？ | ❌ |
| `propagation_border` 在多进程下占多大比例？ | ❌ |

## 2. 设计目标

- **低侵入**：通过宏插桩，不改变任何数值计算路径和 MPI/OMP 执行语义
- **编译期可关闭**：默认 OFF，不影响生产构建的性能和行为
- **两阶段粒度**：
  - Phase 1（主循环级别）：7 个 phase + 总循环时间
  - Phase 2（通信/I/O 内部）：9 个细粒度 sub-phase
- **多进程汇总**：用 `MPI_Reduce` 汇总 avg 和 max，rank 0 输出表格
- **不引入额外 barrier**：不在主循环中加 `MPI_Barrier` 来对齐计时

## 4. Phase 枚举与覆盖范围

```
enum lbm_prof_phase_t {
    // Phase 1 — 主循环
    PROF_SPECIAL_CELLS,           special_cells()
    PROF_COLLISION,               collision()
    PROF_HALO_START,              lbm_comm_halo_exchange_start()
    PROF_PROPAGATION_INTERIOR,    propagation_interior()
    PROF_HALO_FINISH,             lbm_comm_halo_exchange_finish()
    PROF_PROPAGATION_BORDER,      propagation_border()  [仅 nb_y > 1]
    PROF_SAVE_FRAME,              save_frame_all_domain()  [仅写帧步]
    PROF_LOOP_TOTAL,              整体时间步时间（百分比分母）

    // Phase 2 — halo_exchange_start 内部
    PROF_HALO_HORIZ,              水平 blocking send/recv
    PROF_HALO_VERT_PACK,          垂直 send buffer 的 memcpy 打包
    PROF_HALO_VERT_POST,          MPI_Irecv + MPI_Isend 的发起

    // Phase 2 — halo_exchange_finish 内部
    PROF_HALO_WAITALL,            MPI_Waitall 等待
    PROF_HALO_VERT_UNPACK,        垂直 recv buffer 的 memcpy 解包
    PROF_HALO_DIAG,               对角 blocking send/recv

    // Phase 2 — save_frame_all_domain 内部
    PROF_FRAME_BUILD,             build_frame_buf() 本地计算
    PROF_FRAME_MPI,               MPI_Send/Recv
    PROF_FRAME_FWRITE,            fwrite() 写盘
}
```

**嵌套关系**：Phase 2 的 sub-phase 是对应 Phase 1 phase 的子集。三组之和近似等于外层 phase 总时间（误差为函数调用和宏本身的极少开销）：

```
PROF_HALO_START  ≈ PROF_HALO_HORIZ + PROF_HALO_VERT_PACK + PROF_HALO_VERT_POST
PROF_HALO_FINISH ≈ PROF_HALO_WAITALL + PROF_HALO_VERT_UNPACK + PROF_HALO_DIAG
PROF_SAVE_FRAME  ≈ PROF_FRAME_BUILD + PROF_FRAME_MPI + PROF_FRAME_FWRITE
```

## 5. 实现细节

### 5.1 宏接口

```cpp
LBM_PROF_INIT()           // 清零所有累加器，在主循环前调用
LBM_PROF_BEGIN(phase)     // 记录 MPI_Wtime() 到 s_t0[phase]
LBM_PROF_END(phase)       // 累加 elapsed，自增 call_count
LBM_PROF_REPORT()         // MPI_Reduce + rank 0 输出表格
```

关闭时全部展开为 `((void)0)`，无任何运行时开销。

### 5.2 per-rank 数据结构

```cpp
static double s_total[PROF_COUNT];  // 各 phase 累积墙上时间（秒）
static long   s_count[PROF_COUNT];  // 调用次数
static double s_t0[PROF_COUNT];     // 最近一次 begin 的时间戳
```

每个 phase 使用独立的 `t0` 槽，支持嵌套（外层 PROF_LOOP_TOTAL 与内层 phase 同时计时，互不干扰）。

### 5.3 MPI 汇总策略

```cpp
MPI_Reduce(s_total, sum_t, PROF_COUNT, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
MPI_Reduce(s_total, max_t, PROF_COUNT, MPI_DOUBLE, MPI_MAX, 0, MPI_COMM_WORLD);
```

两次 `MPI_Reduce`，每次 `PROF_COUNT * sizeof(double)` = 17 × 8 = 136 字节。开销可忽略（仅在程序结束时执行一次）。**不引入额外 `MPI_Barrier`**；`MPI_Reduce` 本身提供集体同步。

### 5.4 `_start` 内部重构（Pack/Post 分离）

原始代码将 pack 和 Isend/Irecv 交织在一起：

```
pack_bot → Isend_bot → Irecv_top → pack_top → Isend_top → Irecv_bot
```

重构后先完成所有打包，再统一发起请求（先 Irecv 后 Isend，减少缓冲压力）：

```
pack_bot → pack_top → Irecv_top → Irecv_bot → Isend_bot → Isend_top
```

这一顺序调整在数值正确性和 MPI 语义上完全等价（非阻塞请求的发起顺序不影响结果），同时使 `PROF_HALO_VERT_PACK` 和 `PROF_HALO_VERT_POST` 的时间测量更干净。

### 5.5 `save_frame_all_domain` Phase 2 插桩

| rank | PROF_FRAME_BUILD | PROF_FRAME_FWRITE | PROF_FRAME_MPI |
|---|---|---|---|
| rank 0 | 1 次 | comm_size 次（本地 + 每个收到的帧）| comm_size-1 次（每个 MPI_Recv）|
| 其他 rank | 1 次 | 0 次 | 1 次（MPI_Send）|

`avg/call` 使用 rank 0 的本地 call_count 计算，对 rank 0 来说 PROF_FRAME_MPI 的 per-call 时间是"每次 MPI_Recv 耗时"，对其他 rank 是"一次 MPI_Send 耗时"。这是合理的诊断粒度。

单进程路径（`comm_size==1`）仍走 `save_frame()`，不插 Phase 2 宏，避免侵入旧代码路径。

## 7. 输出字段说明

| 字段 | 含义 |
|---|---|
| `avg_total_s` | 该 phase 所有 rank 的累计时间之和 ÷ rank 数 |
| `max_total_s` | 所有 rank 中，该 phase 累计时间最长的那个 rank 的值 |
| `avg/call` | `avg_total_s` ÷ 本 rank（rank 0）的调用次数，单位 ms |
| `pct_loop` | `avg_total_s` ÷ `avg(PROF_LOOP_TOTAL)` × 100% |

**分析并行程序时应优先看 `max_total_s`**，而非 `avg_total_s`：

- `avg` 反映平均资源消耗，适合估算总工作量
- `max` 反映最慢 rank 的实际等待时间，决定端到端性能
- `max - avg` 的差值越大，说明 rank 间负载或网络不均衡越严重
- `PROF_HALO_WAITALL` 的 `max` 远大于 `avg` 是典型通信瓶颈信号

Phase 2 的各行是诊断 Phase 1 行的内部分布，例如：
- `halo_waitall` 的 max 体现 MPI 等待的实际延迟
- `halo_vert_pack` 的比例体现打包开销（集群上应 < 1%）
- `frame: fwrite` 体现磁盘 I/O 的实际耗时

## 8. 使用方法

### 8.1 编译开关

```bash
# 关闭 profiling（默认）
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build

# 开启 profiling
cmake -B build -DCMAKE_BUILD_TYPE=Release -DLBM_ENABLE_PROFILING=ON
cmake --build build
```

`LBM_ENABLE_PROFILING` 通过 `PUBLIC` 传播，自动作用于 `top.lbm-lib`、`top.lbm-exe` 和 `top.display`。

### 8.2 运行

```bash
# 标准运行（profiling 关闭，行为与之前完全一致）
mpirun -np 2 ./build/top.lbm-exe config.txt

# 开启 profiling 后运行
mpirun -np 2 ./build/top.lbm-exe config.txt
# 在 SIMULATION COMPLETED 之后自动追加 profiling 表格输出到 stderr
```

### 8.3 分离 profiling 输出

```bash
mpirun -np 4 ./build/top.lbm-exe config.txt 2>prof.txt
cat prof.txt | grep -A100 "=== LBM Phase Profiling"
```

## 9. 风险与开销

### 9.1 关闭时（默认）

- 宏全部展开为 `((void)0)`
- `profiling.cpp` 中 `#ifdef` 内无任何代码
- 编译器无需生成任何额外指令
- **与原始行为完全一致**

### 9.2 开启时

| 开销来源 | 量化 |
|---|---|
| `MPI_Wtime()` 调用 | 每步约 2×(7+9) = 32 次；MPI_Wtime 典型耗时 < 50 ns，每步总开销 < 2 µs（< 0.01% of ~200 µs/step）|
| 累加操作 | 32 次加法 + 递增，纳秒级 |
| `MPI_Reduce` | 仅在程序结束时执行 2 次，共 272 字节数据，可忽略 |

### 9.3 对计时精度的影响

`MPI_Wtime` 是 wall-clock 时间，由各 rank 独立读取，不需要全局同步。它测量的是：
- 包含 OpenMP 线程内部的并行时间（从调用点到返回点的墙上时间）
- 不包含未被括住的部分（例如 MPI_Barrier 或其他 phase 的时间）

因为 wall-clock 是主线程视角的时间，`PROF_COLLISION` 测量的是 `collision()` 函数的墙上时间，包含了 OpenMP 线程的 fork/join 开销（实际上这正是我们想看到的）。

### 9.4 `Wall-clock != CPU time` 注意事项

在多进程场景下，某些 phase 的 wall-clock 时间会受到其他 rank 的影响：
- `PROF_HALO_WAITALL`：如果某 rank 计算比邻居慢，等待时间会被计入
- `PROF_SAVE_FRAME`（rank 0）：串行收集所有 rank 的帧，时间随进程数线性增长

这正是 profiling 的价值所在——暴露这些隐藏的等待开销。

## 10. 实测验证

### 10.1 测试配置

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release -DLBM_ENABLE_PROFILING=ON
cmake --build build --parallel
```

配置文件：

```txt
iterations           = 20000
width                = 800
height               = 160
obstacle_x           = 100.0
obstacle_y           = 80.0
obstacle_r           = 12.0
reynolds             = 96
inflow_max_velocity  = 0.16
output_filename      = /tmp/prof_out.raw
write_interval       = 99999
```

运行方式：

```bash
OMP_NUM_THREADS=4 mpirun -np 1 ./build/top.lbm-exe /tmp/config_prof.txt
OMP_NUM_THREADS=4 mpirun -np 2 ./build/top.lbm-exe /tmp/config_prof.txt
```

该配置基本等价于"纯计算基线"：`write_interval=99999` 使主循环内几乎不发生写帧，因此 profiling 结果主要反映计算与 halo 通信。

### 10.2 `np=1, OMP=4`

| Phase | avg_total_s | pct_loop | 结论 |
|---|---:|---:|---|
| `collision` | 10.63195 | 65.05% | 主热点 |
| `propagation_interior` | 5.09472 | 31.17% | 第二热点 |
| `special_cells` | 0.60554 | 3.70% | 占比较小 |
| `halo_start` | 0.00515 | 0.03% | 单进程几乎为零 |
| `halo_finish` | 0.00404 | 0.02% | 单进程几乎为零 |
| loop total | 16.34508 | 100.00% | `FOM = 156.16 MLUPS` |

细分 halo：

| Sub-phase | avg_total_s | pct_loop |
|---|---:|---:|
| `halo: horizontal` | 0.00205 | 0.01% |
| `halo: diagonal` | 0.00216 | 0.01% |

结论：

- 单进程下，时间几乎全部花在本地计算；profiling 成功验证当前 MPI 路径对 `np=1` 基本无扰动。
- 下一步若继续优化单进程性能，应优先看 `collision`，其次是 `propagation_interior`。

### 10.3 `np=2, OMP=4`

| Phase | avg_total_s | max_total_s | pct_loop | 结论 |
|---|---:|---:|---:|---|
| `collision` | 5.55306 | 5.55464 | 58.48% | 仍是主热点 |
| `propagation_interior` | 2.81177 | 2.82401 | 29.61% | 第二热点 |
| `halo_finish` | 0.57593 | 0.61218 | 6.07% | 通信等待开始可见 |
| `special_cells` | 0.31290 | 0.31789 | 3.30% | 占比较小 |
| `propagation_border` | 0.15359 | 0.16365 | 1.62% | 仅多进程出现 |
| `halo_start` | 0.08413 | 0.08538 | 0.89% | 发起开销较小 |
| loop total | 9.49523 | 9.51444 | 100.00% | `FOM = 269.05 MLUPS` |

细分 halo：

| Sub-phase | avg_total_s | max_total_s | pct_loop | 结论 |
|---|---:|---:|---:|---|
| `halo: waitall` | 0.48490 | 0.52505 | 5.11% | halo 的主要成本 |
| `halo: vert_unpack` | 0.08796 | 0.09179 | 0.93% | 次要成本 |
| `halo: vert_pack` | 0.07157 | 0.07255 | 0.75% | 次要成本 |
| `halo: vert_post` | 0.00990 | 0.01019 | 0.10% | 很小 |
| `halo: horizontal` | 0.00054 | 0.00055 | 0.01% | 几乎可忽略 |
| `halo: diagonal` | 0.00062 | 0.00063 | 0.01% | 几乎可忽略 |

结论：

- `np=2` 下总性能从 **156.16 MLUPS** 提升到 **269.05 MLUPS**，相对 `np=1` 约 **1.72x**。
- 即使进入多进程，瓶颈仍然是 `collision + propagation_interior`，两者合计约 **88.09%** 的循环时间；说明当前阶段仍以算子优化为主，而不是 MPI 优化为主。
- halo 相关时间中，真正主要的不是消息发起，而是 `MPI_Waitall`；这表明通信成本更多体现为等待邻居完成，而不是 API 调用本身。
- `max_total_s` 与 `avg_total_s` 差距不大，说明 `np=2` 下 rank 间负载较均衡，尚未出现明显失衡。

### 10.4 基于实测数据的下一步优化方向

按优先级排序：

1. `collision`：当前最大热点，优先级最高。
2. `propagation_interior`：第二热点，适合继续做访存与 SIMD 友好性优化。
3. `halo_waitall`：若后续扩展到更多 MPI rank，应重点观察该项是否随进程数快速放大。
4. `special_cells`、`propagation_border`：当前占比较小，不是首要目标。
5. `horizontal` / `diagonal` halo：当前数据表明几乎可忽略，不应作为下一轮主要优化方向。

## 11. 结论

这次改动不是数值优化，而是**可观测性增强**：

- 新增 `profiling.hpp` + `profiling.cpp`（约 120 行，独立模块）
- 在 `main.cpp` 插入 16 个宏调用（Phase 1）
- 在 `communications.cpp` 插入 18 个宏调用（Phase 2），并顺带重构了 `_start` 的 pack/post 顺序（正确性等价，profiling 更干净）
- 同步清理了 `main.cpp` 中已失效的 `temp_render` 死内存分配

开启后能够定量回答"瓶颈在哪个 phase"、"哪个 rank 最慢"、"通信隐藏了多少"等在优化工作中至关重要的问题，为后续在集群环境中的定向优化提供数据基础。
