# 优化22 — `collision_multicell` 显式向量化验证与 `multicell_vec`

**日期：** 2026-04-27  
**分类：** P2 / collision kernel 向量化  
**状态：** 完成；正式 benchmark 通过

---

## 0. 背景与目标

优化21完成了 **SoA multi-cell collision**，正式主路径为 `multicell`。  
其核心收益来自：

- plane pointer hoist
- `j` 内层 stride-1 连续访问
- 更好的标量 ILP

但优化21并没有回答一个关键问题：

> `collision_multicell` 的 `j` 循环到底有没有被编译器自动向量化？

优化22的目标就是：

1. 对 `collision_multicell` 做显式向量化验证
2. 若未向量化，定位根因
3. 在最小改动下修复并引入新路径 `multicell_vec`
4. 正式比较：
   - `multicell`
   - `multicell_vec`

因此，优化22是对优化21的直接延伸，关系如下：

```text
SoA baseline v1.1
  →（优化20）k-outer + baseline
  →（优化20附加实验）k-outer + unrolled
  →（优化21）k-outer + multicell
  →（优化22）k-outer + multicell_vec
```

## 1. 审查结论：`multicell` 并未自动向量化

使用 GCC 15 的向量化报告：

```text
src/lbm/physics.cpp:231:23: missed: couldn't vectorize loop
src/lbm/physics.cpp:232:20: missed: not vectorized: no vectype for stmt: f0_53 = *_52;
  scalar_type: const double
src/lbm/physics.cpp:265:14: missed: not vectorized: complicated access pattern.
```

反汇编确认：

- `j` 循环体中没有 `ymm` / `zmm` 寄存器参与主计算
- 全部为标量 FP 指令：
  - `vaddsd`
  - `vmulsd`
  - `vdivsd`

结论：

- 优化21中的 `multicell` **没有被自动向量化**
- 它的 10–13% 收益完全来自标量优化，而非 SIMD

---

## 2. 根因分析

### 2.1 表面现象

`multicell` 已经做了：

- 18 个 plane pointer hoist（9 读 + 9 写）
- 固定列 `i`
- `j` 方向 stride-1 访问
- plane base pointer 标注 `__restrict__`

按直觉，这样的循环应当具备自动向量化条件。

### 2.2 真正根因

问题出在：**`__restrict__` 没有传递到派生列指针。**

例如：

```cpp
const double* __restrict__ p0 = Mesh_get_plane(mesh_in, 0);
...
const double* cp0 = p0 + col_off;   // 这里 restrict 语义没有自动延续
```

#### OpenMP 闭包机制与 restrict 的关系

`#pragma omp parallel for` 的实现方式是：GCC 将循环体提取为一个独立的内部函数
（mangled 名形如 `_ZL19collision_multicell..._omp_fn.0`），并通过一个结构体
将外部变量”捕获”后传入：

```text
// GCC 等效展开（示意）：
struct omp_data_t {
    double* p0;   // const double* __restrict__ → 退化为普通 double*
    double* p1;
    ...
    double* q0;
    ...
    int W, H;
};

void collision_multicell_omp_fn(void* data_ptr) {
    struct omp_data_t* d = data_ptr;
    // 在这里，d->p0 已经没有 __restrict__ 限定
    // 编译器只看到 d->p0 是一个结构体成员指针，不知道与其他成员不重叠
}
```

关键点：

- C/C++ 规范中 `restrict`（或 `__restrict__`）是**指针变量的局部语义**，
  不是被指向内存的属性
- 当 `p0` 进入 struct 成员后，`restrict` qualifier 不会作为 struct 字段类型的一部分保留
- GCC 闭包内看到的是 `double* d->p0`，与 `double* d->q0` 的别名关系未知
- 因此 GCC 在 `omp_fn` 内对 `cp0 = d->p0 + col_off` 所做的别名分析是保守的：
  无法排除 `cp0` 与 `cq0..cq8` 之间有重叠
- 向量化器遇到不确定的别名即放弃向量化，报告
  `no vectype for stmt: f0 = *cp0; scalar_type: const double`

这就解释了为什么 `p0` 上的 `__restrict__` 对闭包内的向量化**完全无效**，
而在 `cp0` 上显式加 `__restrict__` 才能让闭包内的别名分析重新成立。

因此，优化21的 `multicell` 只是”**SIMD-friendly but not actually SIMD**”。

---

## 3. 修复方案：`collision_multicell_vec`

本轮采用最小改动引入新路径：

- 保留原 `multicell`
- 新增 `multicell_vec`

### 3.1 两个关键修复

#### 修复 1：列指针显式加 `__restrict__`

```cpp
// 修复前
const double* cp0 = p0 + col_off;

// 修复后
const double* __restrict__ cp0 = p0 + col_off;
```

对所有：

- `cp0..cp8`
- `cq0..cq8`

都显式声明 `__restrict__`。

#### 修复 2：对 `j` 循环加 `#pragma GCC ivdep`

```cpp
#pragma GCC ivdep
for (int j = 1; j < H - 1; j++) {
  ...
}
```

目的：

- 明确告知编译器无循环携带依赖
- 配合 `__restrict__` 打通自动向量化条件

### 3.2 新增实现路径

新增：

```cpp
COLLISION_MULTICELL_VEC
```

并将默认路径改为：

```cpp
LBM_COLLISION_IMPL=multicell_vec
```

保留：

- `baseline`
- `unrolled`
- `multicell`

作为对照路径。

## 4. 向量化验证结果

修复后 GCC 向量化报告：

```text
src/lbm/physics.cpp:337:19: optimized: loop vectorized using 64 byte vectors
src/lbm/physics.cpp:337:19: optimized: loop vectorized using 32 byte vectors
src/lbm/physics.cpp:337:19: optimized: loop vectorized using 16 byte vectors
```

### 三级路径的含义

GCC 对同一个 j 循环实际生成**四个版本**，按 H 的可用元素数依次选择：

```text
[j=1 到 j=H-2] 中的 H-2 个 cell（此处 H=162，H-2=160）

  版本 1（主路径）：
    每迭代处理 8 个 cell（64 byte = 8 × 8 byte double）
    使用 zmm 寄存器
    处理：j = 1, 9, 17, ..., 153（共 19 × 8 = 152 个 cell）

  版本 2（回退路径 1）：
    每迭代处理 4 个 cell（32 byte = 4 × 8 byte double）
    使用 ymm 寄存器
    处理：版本 1 结束后剩余的 4 个 cell（157..160）

  版本 3（回退路径 2）：
    每迭代处理 2 个 cell（16 byte = 2 × 8 byte double）
    使用 xmm 寄存器
    处理剩余 2 个 cell

  版本 4（标量收尾）：
    处理最后 1 或 0 个 cell
```

对于 H-2=160（本工程默认），160 = 20 × 8，整除，全部走版本 1（20 次 zmm 迭代），
其他版本在运行时不会执行但仍被生成（编译期不知道 H 的具体值）。

np=4 时 H=42，H-2=40 = 5 × 8，同样整除，每列 5 次 zmm 迭代——迭代次数少，
版本切换和循环控制开销比例更高，这是 np=4 向量化收益（2×）低于 np=1（4.5×）的直接原因。

反汇编确认核心指令为：

```text
vmovupd (%r10,%rax,1),%zmm23   // cp0[j:j+7]
vmovupd (%r14,%rax,1),%zmm19   // cp1[j:j+7]
...（9 planes 各 1 条 vmovupd zmm）
vaddpd  %zmm19,%zmm23,%zmm2    // density accumulation
vdivpd  %zmm2,%zmm27,%zmm3     // 8 cells / 1 vector division
...（后续 vmulpd + vaddpd）
vmovupd %zmm1,(%r8,%rax,1)     // 写回 cq0[j:j+7]
...（9 output planes 各 1 条 vmovupd zmm）
```

结论：

- `multicell_vec` 的 `j` 循环已经被真正向量化
- SIMD lane 对应多个连续 cell（`j` 方向），而不是多个方向

## 6. 分析

### 6.1 为什么 AVX-512 没有被除法拖垮

直觉上会担心：

- `vdivpd zmm` 吞吐量低
- 向量除法可能不如多个标量除法

但在当前平台上，8 个 double 的向量除法仍然优于标量并行：

| 方案 | 8 个 double 除法 | 每 double CPI |
| --- | ---: | ---: |
| 8 scalar `divsd`（2 div units） | 52 CPI | 6.5 |
| `vdivpd zmm` | 24 CPI | 3.0 |
| 向量除法更快 | **2.2×** | — |

**推导说明（Zen 4 实测吞吐量，来自 Agner Fog 指令表）：**

- `divsd`：throughput = 13 CPI（非流水线，每次除法独占除法单元 13 周期）
- Zen 4 有 **2 个独立 FP 除法单元**，可同时执行 2 条独立的 `divsd`
- 8 个独立 `divsd`：分为 4 组，每组 2 条并行 → 4 × 13 = **52 CPI**
- `vdivpd zmm`（8 doubles）：throughput = 24 CPI（同样使用两个除法单元，
  但以向量方式执行，64 byte / 24 CPI = **3.0 CPI per double**）
- 结论：向量除法比标量并行快约 **2.2×**（52 / 24）

注意：此结论与很多"标量除法更快"的直觉相反。反直觉的关键是：
两个除法单元可以同时执行 `vdivpd zmm` 的不同部分（低 4 lane + 高 4 lane），
而 8 个独立 `divsd` 也只能用同样的 2 个单元、分 4 批执行，并没有额外优势。

同时，其余部分：

- density
- vx / vy
- equilibrium

都能被 8-lane 向量化进一步摊薄，所以整体 `collision` 加速可达：

- `np=1`: 约 4.5×
- `np=2`: 约 4.0×
- `np=4`: 约 2.0×

### 6.2 瓶颈迁移

`collision` 被大幅压缩后，瓶颈明显转移：

- `np=1`：`propagation_interior` 占比显著上升
- `np=2`：`collision` 与 `halo_finish` 已接近
- `np=4`：`halo_finish` 成为主瓶颈，尤其是：
  - `vert_pack`
  - `vert_unpack`

这意味着优化22之后，继续深挖 `collision` 的边际收益已经开始下降。

## 8. 结论与处置

| 问题 | 结论 |
| --- | --- |
| `multicell` 是否已自动向量化？ | **否**；全部标量执行 |
| 根因 | 列指针缺 `__restrict__`，GCC 别名分析无法证明 `cp` vs `cq` 不重叠 |
| 修复方案 | `__restrict__` on `cp/cq` + `#pragma GCC ivdep` |
| `multicell_vec` 是否向量化？ | **是**；AVX-512 64-byte，含 `vdivpd zmm` |
| 显式 AVX 是否值得保留？ | **是**；`np=1 +2.58×`，`np=2 +1.97×`，`np=4 ~+20%` |
| 新默认路径 | **`multicell_vec`** |
| 旧 `multicell` 路径 | 保留作为标量对照（`LBM_COLLISION_IMPL=multicell`） |

最终结论：

- 优化22不是“继续微调 multicell”，而是完成了 **真正的显式向量化验证**
- `collision_multicell` 原本并未自动向量化
- `multicell_vec` 通过最小修复成功打通 AVX-512 自动向量化
- 向量化后的收益足够大，值得单独作为一轮正式优化记录

下一步优先级应转向：

1. `halo_finish` 内部的 vertical pack/unpack 格式改造
2. `np=1` 下 `propagation_interior` 的进一步压缩
