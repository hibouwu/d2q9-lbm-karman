# 优化06 — 删除 halo_exchange 中重复通信阶段（P1-3）

**日期：** 2026-04-22  
**分类：** P1 / 冗余通信消除  
**状态：** 完成，正确性已验证

---

## 1. 问题描述

原始 `lbm_comm_halo_exchange` 中存在两处重复通信阶段：

### 重复 1：Bottom-left 对角通信出现两次

```cpp
// 第一次（约 317-319 行）
lbm_comm_sync_ghosts_diagonal(COMM_SEND, CORNER_BOTTOM_LEFT, x=1, y=height-2)
lbm_comm_sync_ghosts_diagonal(COMM_RECV, CORNER_TOP_RIGHT,   x=width-1, y=0)
MPI_Barrier(...)

// 第二次（约 329-331 行，完全相同）
lbm_comm_sync_ghosts_diagonal(COMM_SEND, CORNER_BOTTOM_LEFT, x=1, y=height-2)
lbm_comm_sync_ghosts_diagonal(COMM_RECV, CORNER_TOP_RIGHT,   x=width-1, y=0)
MPI_Barrier(...)
```

### 重复 2：Right-to-left 水平通信出现两次

```cpp
// 第一次（约 287-289 行）
lbm_comm_sync_ghosts_horizontal(COMM_SEND, left_id, x=1)
lbm_comm_sync_ghosts_horizontal(COMM_RECV, right_id, x=width-1)
MPI_Barrier(...)

// 第二次（约 347-349 行，完全相同，无 barrier）
lbm_comm_sync_ghosts_horizontal(COMM_SEND, left_id, x=1)
lbm_comm_sync_ghosts_horizontal(COMM_RECV, right_id, x=width-1)
```

### 影响量化

| 重复通信 | 类型 | 每次 halo_exchange 额外通信量（np=2, width=800）|
|---|---|---|
| Bottom-left 重复 | 对角 1 格点（通常 target=-1，skip） | 0（目标为 -1 时直接返回）|
| Right-to-left 重复 | 水平列（nb_x=1 时 target=-1，skip） | 0 |

> 对于常用的 2 的幂次进程数（pgcd(np,800) = np，nb_x=1），所有水平和对角通信的 target_rank = -1，函数直接返回。因此重复阶段的实际通信量为零，但每次调用的函数进入开销（条件判断 + 函数调用）以及额外 barrier（P1-1 中已删除）仍有小量开销。

---

## 2. 修改内容

与 P1-1 (优化04) 同批完成，在重写 `lbm_comm_halo_exchange` 时一并删除：
- 第二个 "Bottom left phase" 及其 barrier
- 末尾重复的 "Right to left phase"

最终函数包含 8 个通信阶段（正确的不重复版本），无任何 barrier。

---

## 3. 正确性验证

`python3.12 -m lbm_viz --check np1.raw np2.raw` → `ok`

对于 nb_x=1 的常用配置，重复阶段目标均为 -1（skip），删除后不影响任何数据传输。

---

## 4. 结论

- 消除 2 个重复通信阶段 + 1 个相关 barrier（总 barrier 由 9→8，P1-1 再全删）
- 对常用进程数（2 的幂次）无实际通信影响，但清理了代码逻辑
- 为非 2 的幂次进程数（如 np=3，nb_x=3）提前消除潜在的重复数据覆盖风险
