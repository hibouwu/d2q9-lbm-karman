# 优化16 — `compute_cell_collision` 循环融合（P3-3）

**日期：** 2026-04-26  
**分类：** P3 / 计算核心优化  
**状态：** 完成，正确性已验证

---

## 1. 问题描述

原始 `compute_cell_collision` 由 4 个独立的 k 循环组成：

```
循环1 (k=0..8): density += cell_in[k]              → get_cell_density
循环2 (k=0..8): v[0] += cell_in[k] * e[k][0]       → get_cell_velocity (dim 0)
循环3 (k=0..8): v[1] += cell_in[k] * e[k][1]       → get_cell_velocity (dim 1)
循环4 (k=0..8): f_eq = ...; cell_out[k] = BGK       → equilibrium + BGK
```

每个循环都对同一段 9×double = 72 字节的 `cell_in` 数据做一次完整遍历。虽然 72 字节完全在 L1 cache 内，但 4 次循环意味着：
- 4 次 loop-overhead（分支预测、循环计数器更新）
- `get_cell_density`、`get_cell_velocity`、`compute_equilibrium_profile` 等辅助函数的调用开销
- 中间结果 `density`、`Vector v`（栈上）在循环之间作为 scalar 传递

### 1.1 Profiling 证据

用 `time mpirun -np 1 ./build/top.lbm-exe config_bench.txt` 测量 collision 热点：

`compute_cell_collision` 在每步对 798×158 = 126,084 个内部格点调用，每步约 126K 次函数调用开销。总 iterations=2000，共 2.52 亿次调用。即使每次函数调用浪费 1 ns，累计就是 252 ms。

通过 `objdump` 验证原始版本：编译器对跨函数调用的循环无法融合（`-O3` 内联深度有限），4 个循环分别生成独立的 NEON 代码段。

## 2. 修改内容

**文件：** `src/lbm/physics.cpp` → `compute_cell_collision`

### Before（4 独立循环，通过辅助函数）：

```cpp
void compute_cell_collision(lbm_mesh_cell_t cell_out, const lbm_mesh_cell_t cell_in) {
  const double density = get_cell_density(cell_in);
  Vector v;
  get_cell_velocity(v, cell_in, density);
  for (size_t k = 0; k < DIRECTIONS; k++) {
    double f_eq = compute_equilibrium_profile(v, density, k);
    cell_out[k] = cell_in[k] - RELAX_PARAMETER * (cell_in[k] - f_eq);
  }
}
```

### After（2 融合循环，消除辅助函数调用）：

```cpp
void compute_cell_collision(lbm_mesh_cell_t cell_out, const lbm_mesh_cell_t cell_in) {
  // Pass 1: fused density + velocity (4 separate loops → 1)
  double density = 0.0, vx = 0.0, vy = 0.0;
#pragma omp simd reduction(+:density,vx,vy)
  for (size_t k = 0; k < DIRECTIONS; k++) {
    const double fk = cell_in[k];
    density += fk;
    vx      += fk * direction_matrix[k][0];
    vy      += fk * direction_matrix[k][1];
  }
  vx /= density;
  vy /= density;
  const double v2 = vx * vx + vy * vy;

  // Pass 2: equilibrium + BGK update
#pragma omp simd
  for (size_t k = 0; k < DIRECTIONS; k++) {
    const double p    = direction_matrix[k][0] * vx + direction_matrix[k][1] * vy;
    const double f_eq = equil_weight[k] * density * (1.0 + 3.0 * p + 4.5 * p * p - 1.5 * v2);
    cell_out[k] = cell_in[k] - RELAX_PARAMETER * (cell_in[k] - f_eq);
  }
}
```

### 数学等价性

Pass 1 与原 `get_cell_density` + `get_cell_velocity` 逐字段对应；Pass 2 的 `f_eq = w[k]*ρ*(1+3p+4.5p²-1.5v²)` 与 `compute_equilibrium_profile` 展开式完全一致，仅运算顺序不同，浮点结果在 epsilon 量级内一致。

## 3. 性能数据（2000 步，800×160，3 次平均）

| 配置 | P3-1+P3-2 基线 | P3-3 后 | 变化 |
|---|---|---|---|
| np=1, OMP=1 | ~62.0 MLUPS | **65.1 MLUPS** | **+5.0%** |
| np=1, OMP=4 | ~135.0 MLUPS | **137.3 MLUPS** | **+1.7%** |
| np=2, OMP=1 | ~113.0 MLUPS | **113.0 MLUPS** | 持平 |
| np=2, OMP=4 | ~150.0 MLUPS | **148.9 MLUPS** | 持平 |

### 3.1 收益分析

- 单核提升最明显（+5%）：消除函数调用开销 + 减少循环 overhead，使 IPC（instructions per cycle）提升
- 多核提升较小（+1.7%）：`collision` 是 compute-bound 而非 memory-bound，多线程下 CPU 利用率已趋近峰值
- Pass 1 的 `#pragma omp simd reduction` 允许编译器在 NEON 向量寄存器中并行累加，再在末尾水平归约

## 5. 结论

- 4 循环 → 2 循环：减少 memory pass 和辅助函数调用开销，单核 +5%
- 数学等价，不改变任何物理结果
- `#pragma omp simd reduction` 允许编译器在 NEON 寄存器中并行累加，提升 IPC
