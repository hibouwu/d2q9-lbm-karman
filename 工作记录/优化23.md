# 优化23：FMA 与 prefetch 实验评估

## 1. 背景

`collision_multicell_vec`（优化21 的最终路径）已在 Zen 4 上实现 AVX-512 自动向量化，每迭代生成 8-wide `zmm` 指令。本轮实验评估两个潜在的进一步优化：

1. **FMA**：乘加融合（`vfmadd*pd zmm`），减少指令数和舍入误差。
2. **软件预取**：手动插入 `__builtin_prefetch`，覆盖硬件流预取器可能遗漏的第 9 条读流。

**基准配置**（`bench_config.txt`）：
```
iterations=20000, width=800, height=160, write_interval=99999
```
OMP_NUM_THREADS 按 `np` 等分物理核数（机器 8 核），避免超额认购：np=1 用 4T，np=2 每进程 4T，np=4 每进程 2T。

---

## 2. FMA 审计（否定实验）

### 2.1 方法

用 `objdump -d` 提取 `collision_multicell_vec` 的 OMP 内部函数（`_omp_fn.0`）并统计 `vfmadd` 指令数：

```bash
objdump -d build/lib/libtop.lbm-lib.so \
  | awk '/collision_multicell_vec.*omp_fn/{found=1} found && /ret/{found=0} found' \
  | grep -c vfmadd
```

**结果：92 条 `vfmadd*pd zmm` 指令。**

典型片段：

```asm
vfmadd231pd %zmm0,%zmm0,%zmm5   # density += cp_k[j] 的累加融合
vfmadd132pd %zmm1,%zmm2,%zmm3   # equilibrium 计算中的 a*b+c
```

### 2.2 结论

GCC 在 `-O3 -march=native`（Zen 4 支持 AVX-512）下已全量自动生成 FMA。`collision_multicell_vec` 的乘加序列 100% 融合，**无需手动干预**。

## 3. 软件预取实验

### 3.1 理论动机

`collision_multicell_vec` 的内层 j 循环对 9 个 input plane（`cp0..cp8`）做顺序读取，每个 plane 内步长为 1 double（stride-1）。

Zen 4 L1D 流预取器容量约 8 条独立流。9 条读流 > 8，理论上第 9 条 plane 每列开头持续 L2 miss。

- 每列工作集：162 doubles/plane × 8 B × 9 = 11.7 KB < L1D（32 KB）
- np=1 每列 zmm 迭代数：(162−2)/8 = 20 次
- np=4 每列 zmm 迭代数：(42−2)/8 = 5 次
- Zen 4 L2 延迟：≈12 cycles；`vdivpd zmm` 吞吐：24 CPI

预取距离 DIST=8（1 次 zmm 迭代 = 1 cache line）理论上足以覆盖 L2 延迟。

### 3.2 实现：`collision_multicell_vec_pf`

在 `collision_multicell_vec` 的 j 循环中对 9 个 input plane 插入软件预取（`DIST=8`，只读，L1 locality）：

```cpp
#pragma GCC ivdep
for (int j = 1; j < H - 1; j++) {
    __builtin_prefetch(cp0 + j + LBM_PREFETCH_DIST, 0, 3);  // × 9 planes
    __builtin_prefetch(cp1 + j + LBM_PREFETCH_DIST, 0, 3);
    // ...
    __builtin_prefetch(cp8 + j + LBM_PREFETCH_DIST, 0, 3);
    // 原 collision 计算...
}
```

`LBM_PREFETCH_DIST` 默认 8，可通过编译宏覆盖。通过 `LBM_COLLISION_IMPL=multicell_vec_pf` 环境变量选择。

### 3.3 汇编验证

```bash
objdump -d build/lib/libtop.lbm-lib.so | grep prefetch | wc -l
```

`multicell_vec` 的 OMP closure 中：0 条 prefetch 指令（纯依赖硬件）。
`multicell_vec_pf` 的 OMP closure 中：9 条 `prefetcht0` 指令出现在每次循环迭代内。

## 5. 分析

### 5.1 FMA：无需优化

编译器已自动融合，92 条 `vfmadd*pd zmm` 覆盖所有乘加序列。显式 FMA 实验无意义，跳过。

### 5.2 软件预取：负收益

prefetch 版本在 collision 阶段一致变慢（np=1 +4%，np=2 +4.5%，np=4 +9%），总体 FOM 随 np 增大而劣化更显著。原因分析：

**开销超过收益**。每次 j 循环迭代增加 9 条 `prefetcht0` 指令。对于列长较短的 np=4（每列 5 次 zmm 迭代），预取指令的固定开销占比更高。

**硬件预取器表现更好**。Zen 4 的实际流预取容量可能大于标称 8 条，或者列间切换代价较低（11.7 KB 工作集远小于 L1D 32 KB，多列共享 cache 行）。9 条读流并未造成理论预期的显著 miss。

**向量化影响**。`__builtin_prefetch` 出现在 `#pragma GCC ivdep` 循环体内，可能干扰自动向量化的迭代分组，导致编译器降级部分迭代到 ymm/scalar 路径。

**halo waitall 变长**：prefetch 版 collision 更慢 → MPI 进程间时间差拉大 → waitall 同步等待增加，放大了 np>1 下的劣化。

### 5.3 np 越大，劣化越严重

每进程列高缩短（np=4 时 H=42，每列 5 次 zmm 迭代），prefetch 的固定指令开销相对计算量占比越高；同时 halo waitall 放大效应在多进程下更明显。

---

## 6. 结论

| 实验 | 结论 |
|---|---|
| FMA 审计 | 已自动生成，92 条 `vfmadd*pd zmm`，无需干预 |
| 软件预取 DIST=8 | 全配置负收益或中性，**放弃** |

**默认路径维持 `collision_multicell_vec`，`multicell_vec_pf` 作为已验证的失败实验路径保留在代码库。**
